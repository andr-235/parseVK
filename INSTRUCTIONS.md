# ParseVK — Instructions

Платформа социальной аналитики для мониторинга и выявления противоправных высказываний в постах и комментариях. Парсит ВКонтакте, Telegram и WhatsApp.

**Default branch:** `fastapi-microservices-rewrite` (проект в процессе перехода с Node.js/Prisma на FastAPI-микросервисы)

---

## Архитектура

Микросервисная архитектура (8 FastAPI-сервисов) + React-фронтенд. Асинхронная коммуникация — Kafka (event-driven), синхронная — HTTP через API Gateway.

```
Frontend (React/Vite) → API Gateway → Identity / Tasks / Content / Moderation / Telegram / IM / VK Services
                                                                                               ↕
                                                                                             Kafka
```

**Kafka-топики:** `parsevk.tasks.events`, `parsevk.vk.events`, `parsevk.im.events`

**PostgreSQL:** 6 отдельных баз (`identity`, `tasks`, `vk`, `content`, `moderation`, `im`)

## Структура микросервиса

Каждый сервис следует трёхслойной архитектуре **Router → Service → Repository**:

```
services/<name>/
  app/
    core/       # config (pydantic-settings), dependencies, exceptions
    modules/    # feature-модули (router, service, repository, schemas)
    db/         # SQLAlchemy models, session factory
    main.py     # FastAPI app factory / create_app()
  alembic/      # миграции (если есть БД)
  alembic.ini   # конфиг миграций
  tests/
  pyproject.toml
  Dockerfile
  uv.lock
```

Все сервисы билдятся через **Hatchling** (`hatchling.build`). Общая библиотека — `libs/py/common/` (shared models, exceptions, kafka helpers).

### Docker-образы

Базовые образы:
- **Python-сервисы:** `python:3.12.13-slim` + `uv:0.11.6` (multi-stage)
- **Фронтенд (build):** `oven/bun:1-alpine` → **продакшн:** `nginx:alpine`

## UI/UX: человекоцентричные правила

ParseVK — рабочий инструмент оператора-аналитика. Каждый пиксель либо помогает принять решение, либо мешает. Эти правила основаны на исследованиях NN/g, WCAG 2.2 и практиках enterprise-продуктов.

### 1. Дизайн для решения, не для данных

Каждый элемент на экране должен отвечать на вопрос оператора. Если метрика или виджет не влияют на решение — удалите.

- **3-секундное сканирование:** пользователь должен понять ситуацию и следующее действие за 3 секунды. Если нет — упрощайте.
- **Одна задача на экран:** экран решает одну задачу. Всё остальное — прогрессивное раскрытие.
- **Не путать данные с инсайтами:** график ради графика — шум. Каждый график отвечает на конкретный вопрос («растёт ли число нарушений?»).

### 2. Иерархия: глаз видит главное первым

```
[ KPI #1 ] [ KPI #2 ] [ KPI #3 ]               <- 3-5 ключевых метрик
------------------------------------------------
[ Тренды / графики ]                            <- контекст, динамика
------------------------------------------------
[ Таблицы / детали ]                            <- данные для анализа
```

- **F-паттерн:** самое важное — в левом верхнем углу. Взгляд движется слева направо, сверху вниз.
- **Не более 5-7 KPI на экран:** когнитивный лимит. Если метрик больше — группируйте или прячьте за раскрытие.
- **Крупнее = важнее:** визуальный вес элемента (размер, жирность, контраст) соответствует его значимости.

### 3. Контекст: число без контекста — не число

Любая метрика должна сопровождаться хотя бы одним из:
- **Сравнение с прошлым периодом:** «+12% к прошлой неделе»
- **Цель/норма:** «78% при норме 80%»
- **Тренд:** спарклайн или стрелка направления
- **Статус:** зелёный = норма, жёлтый = внимание, красный = критично

Число без контекста заставляет пользователя запоминать или искать норму — это лишняя когнитивная нагрузка.

### 4. Цвет — только для состояния

- Зелёный (`oklch 0.55 0.13 150`) — чисто, норма, успех
- Жёлтый/amber (`oklch 0.65 0.14 85`) — внимание, требуется проверка
- Красный (`oklch 0.5 0.18 30`) — нарушение, критично, ошибка
- Синий/primary (`oklch 0.55 0.12 230`) — действие, активное состояние

Правила:
— Не подсвечивать всё подряд. Если всё зелёное — цвет перестаёт нести информацию.
— Один accent-цвет на экран. Второй цвет — только для семантического контраста.
— Цвет никогда не используется для декора. Никаких «красивых» радужных палитр.

### 5. Прогрессивное раскрытие

Пользователь видит ровно столько информации, сколько нужно сейчас. Остальное — по запросу.

| Уровень | Что показывает | Пример |
|---|---|---|
| Primary | Сводка, 3-5 метрик, статусы | KPI-карточки, сводка задач |
| Secondary | Контекст, тренды, сравнения | Графики, распределения |
| Territory | Детали, сырые данные | Таблицы, лог событий, Drill-down |

- **Сначала summary, потом details.** Никогда не наоборот.
- **Drill-down:** клик по KPI → открывает график → клик по графику → открывает таблицу транзакций.
- **Smart defaults:** показывайте данные «за последние 30 дней» сразу. Не заставляйте выбирать даты, чтобы увидеть хоть что-то.

### 6. Скорость — это UX

| Метрика | Цель |
|---|---|
| Initial load | < 2 секунды |
| Data refresh | < 500 мс |
| Interactivity response | < 150 мс |

- **Skeleton screens** вместо спиннеров — показывайте каркас, даже если данные ещё грузятся.
- **Lazy load:** загружайте верх экрана первым. Вторичные метрики — по скроллу.
- **Кэшируйте агрегаты:** не перезапрашивайте одно и то же. staleTime в TanStack Query — ваш друг.
- **Асинхронная загрузка:** не блокируйте интерфейс ожиданием тяжёлого запроса.

### 7. Доступность (Accessibility)

- **Контраст:** минимум 4.5:1 для текста, 3:1 для крупных элементов (WCAG AA).
- **Клавиатурная навигация:** все действия доступны с клавиатуры (Tab, Enter, Escape, стрелки).
- **Не полагаться на цвет:** статусы дублировать иконками или текстом. Пользователь с дальтонизмом должен понимать интерфейс.
- **Touch targets:** минимум 44x44 px на мобильных устройствах.
- **ARIA-метки:** для screen reader'ов. Все интерактивные элементы имеют понятное описание.

### 8. Работа с данными

**Таблицы:**
- Фиксированный заголовок при скролле
- Сортировка по всем колонкам (один клик — сортировка, второй — обратная)
- Фильтрация: строка поиска + быстрые фильтры
- Пагинация или виртуальный скролл для >100 строк
- Inline editing для быстрых изменений
- Bulk actions (выбрать несколько → действие)

**Графики:**
- Bar chart — для сравнения категорий
- Line chart — для трендов во времени
- Bar всегда с нуля. Усечённая ось Y искажает восприятие.
- Pie chart — только для 2-3 сегментов и с подписями значений. Для большего — bar chart.
- НЕ использовать: 3D-графики, donut-чарты с 5+ сегментами, area chart без нулевой базы
- Прямые подписи вместо легенды — не заставляйте глаз прыгать между графиком и легендой

**Фильтры:**
- Глобальные фильтры — меняют все виджеты на экране
- Сохранённые пресеты — «Последняя неделя», «Этот месяц» — одним кликом
- Прогрессивная сложность: базовый фильтр для всех, advanced — для power users
- Показывать активные фильтры — пользователь всегда видит, какие фильтры применены

### 9. Сообщения и обратная связь

- **Успех:** тост или inline-уведомление. Исчезает само через 3-5 секунд.
- **Ошибка:** конкретное сообщение, что пошло не так и что делать. Не «Ошибка 500», а «Не удалось загрузить данные. Попробуйте обновить страницу».
- **Пустое состояние:** не «Нет данных», а объяснение + действие. «Задачи не найдены. Создайте первую задачу».
- **Загрузка:** skeleton screen или прогресс-бар. Не моргающий и не дёргающийся интерфейс.
- **Подтверждение:** только для опасных действий (удаление, массовая операция). Не спрашивать «Вы уверены?» на каждое действие.

### 10. Иконки (lucide-react)

Проект использует **lucide-react** — стандарт индустрии (5M загрузок/нед), дефолтная библиотека shadcn/ui.

Почему lucide-react:
- 1500+ иконок в едином stroke-стиле на сетке 24x24
- Минимальный размер бандла (~0.5 KB/иконка при tree-shaking)
- Настраиваемый `strokeWidth` (1.5 для лёгкого, 2.5 для жирного)
- `createLucideIcon` для кастомных иконок в едином стиле
- ISC-лицензия, активное сообщество

Правила использования:
- Иконка всегда сопровождает текст (кнопки, пункты меню), но не заменяет его
- Размер: 16px для inline, 20px для кнопок, 24px для пустых состояний
- `strokeWidth={1.5}` для декоративных, `strokeWidth={2}` для интерактивных
- Цвет иконки наследует `currentColor` — никогда не красить напрямую
- Не использовать декоративные иконки без информации — каждая иконка несёт смысл

### 11. Микро-взаимодействия

- **Transition:** 150ms ease-out для всех состояний (hover, focus, open/close)
- **Никаких bounce/elastic:** только линейные или ease-out кривые
- **Hover:** только изменение фона (neutral-hover). Никаких scale, shadow, lift.
- **Focus:** изменение border-color на primary. Никаких glow, ring-offset.
- **Нажатие (active):** мгновенный отклик (50ms) — смена фона или opacity

### 12. Навигация и ориентация

- **Пользователь всегда знает, где он:** хлебные крошки или активный пункт меню.
- **Не более 5-7 пунктов в основной навигации.** Если больше — группируйте.
- **Сайдбар:** 224px, label-стиль (uppercase, 12px, letter-spacing 0.05em), иконка + текст.
- **Поиск по навигации** — для больших систем (Command + K).
- **Хлебные крошки** — в глубине (3+ уровня).

### 13. Терминология и копирайтинг

- **Человеческий язык:** не «Создать объект задачи парсинга», а «Новая задача». Не «Детализированная дельфабрикация» — пользователь не должен расшифровывать интерфейс.
- **Однозначные CTA:** не «Отправить», а «Запустить парсинг». Не «ОК», а «Сохранить» или «Удалить».
- **Единая терминология:** «Автор» везде автор, а не то «Пользователь», то «Создатель», то «Владелец».
- **Метки полей:** кратко и понятно. Не более 3-4 слов. Если нужно объяснение — tooltip с иконкой.
- **Числа и даты:** единый формат everywhere. `15.03.2026` или `2026-03-15` — один раз и везде. Разделители тысяч — не пробелы, а узкие пробелы или ничего.

### 14. Empty states

Пустой экран — не баг, а дизайн-задача. Хороший empty state:
1. **Объясняет** — «У вас пока нет задач»
2. **Предлагает действие** — «Создайте первую задачу»
3. **Убирается** после первого действия — не показывается снова

Плохой empty state: пустая таблица, серый экран, «No data» без контекста.

### 15. Mobile

Даже для enterprise-инструмента:
- **Mobile-specific view**, не респонсивное сжатие десктопа
- 3-5 ключевых метрик крупными карточками
- Спарклайны вместо сложных графиков
- Tap-to-reveal вместо hover
- Touch targets >= 44px
- Вертикальный скролл вместо сетки

## Антипаттерны (запрещено)

### Архитектура и код

- **Бизнес-логика в Router или Repository** — трёхслойная архитектура (Router → Service → Repository). Router только маршрутизирует, Repository только читает/пишет, Service содержит бизнес-логику.
- **Толстые сервисы** — сервис не должен делать всё. Разбивайте на модули по фичам (`app/modules/`).
- **Файлы > 100-150 строк — запрещены.** Оптимальный размер файла: 100-150 строк. Если файл превышает лимит — требует декомпозиции: разбить на модули, выделить утилиты, вынести типы/схемы. После дробления — обязательное code review и рефакторинг. Исключение: файлы конфигурации (`pyproject.toml`, `docker-compose.yml`), миграции alembic, autogenerated-код.
- **Синхронные цепочки сервисов** — сервис A → сервис B → сервис C по HTTP. Используйте Kafka для асинхронных сценариев, HTTP только для прямых запросов (CQRS).
- **Shared database между сервисами** — каждый сервис владеет своей БД. Доступ к чужим данным только через API сервиса-владельца.
- **Игнорирование edge cases** — обрабатывайте: пустые списки, null-поля, дубликаты, невалидные ID, таймауты, отсутствие данных.
- **Тихие ошибки** — никогда не используйте `except: pass`. Всегда логируйте причину. Не swall'те исключения без обработки.
- **Нет идемпотентности Kafka-consumer'ов** — повторная доставка событий штатна. Consumer должен быть идемпотентным (dedup по event_id или upsert).
- **Магические числа и хардкод** — все константы и пороги в конфиг или переменные окружения. Никаких `if status == 3` — используйте Enum.
- **Нет типов** — Python-код обязан иметь type hints. Pydantic схемы для всех входящих/исходящих данных. Pydantic Settings для конфигов.
- **Смерджено, но сломано** — мёржить можно только после: локальные тесты пройдены, CI зелёный, нет конфликтов, секреты не попали в коммит.

### Фронтенд

- **Состояние в нескольких местах** — TanStack Query для серверного состояния, Zustand для клиентского. Не дублировать одно и то же в React state + store + query cache.
- **Нет loading/error состояний** — каждый запрос должен обрабатывать `isLoading`, `isError`, пустые данные. Показывать скелетоны/заглушки.
- **Избыточные запросы** — не фетчить то, что уже есть. Используйте query key, staleTime, cache. Избегайте overfetching (запрашивайте только нужные поля).
- **Modal как первое решение** — для деталей используйте Detail Panel (правый drawer), а не модалку. Модалка — только для confirm/action.

### Git и процесс

- **Одна задача = один PR** — не смешивать несвязанные фичи, багфиксы и рефакторинг в одном PR.
- **PR из default branch** — ветки только от issue: `<type>/<issue-number>-<summary>`. Никогда не создавать PR из default branch.
- **Self-approve** — нельзя аппрувить свои PR. Ревью — отдельный человек или AI-агент.
- **Деструктивные git-команды без спроса** — `git reset --hard`, `git rebase`, `git push --force` только с явного разрешения.
- **Коммит секретов** — `.env`, `*.pem`, `*.key`, токены, пароли никогда не попадают в git.
- **MVP/заглушки** — не коммитить «TODO-реализую позже», заглушки, `pass` вместо реализации. Production-ready качество.

### Дизайн (см. также `DESIGN.md`, `PRODUCT.md`)

- **Клоунские цвета** — никаких кислотных, неестественных, игрушечных палитр.
- **Hero-секции и маркетинг** — это рабочий инструмент, не лендинг. Никаких декоративных блоков.
- **Терминал-стиль** — никакой стилизации «под консоль» (зелёный текст на чёрном).
- **Gradient text / glassmorphism** — запрещены оба приёма.
- **Card-верстка по умолчанию** — основной контейнер — таблицы и плоские секции. Карточки только где явно нужно.
- **Bounce/elastic анимации** — только 150ms ease-out transition.
- **Display-шрифты в UI-элементах** — Space Grotesk для всего, Geist Mono только для ID/кода/меток.

## Troubleshooting

**Проблема:** `docker compose up` не стартует сервисы
**Решение:** Проверьте, что создан `.env` и заполнены обязательные переменные. Проверьте логи: `docker compose logs <service>`.

**Проблема:** VK API возвращает ошибки
**Решение:** Убедитесь, что `VK_TOKEN` валидный и имеет права wall, groups, users, offline. Получить: https://vkhost.github.io

**Проблема:** Не запускаются миграции БД
**Решение:** Проверьте, что БД поднята (`docker compose ps`) и переменные `*_DATABASE_URL` корректны.

**Проблема:** CI падает с ошибками про `api/` или Prisma
**Решение:** CI-файл (`.github/workflows/ci.yml`) устарел — ссылается на старый Node.js-бэкенд. FastAPI-сервисы пока не добавлены в CI. Нужно обновить.

**Проблема:** parsevkctl не может создать ветку
**Решение:** Убедитесь, что default branch в `tools/parsevkctl-go/config.json` указан как `fastapi-microservices-rewrite`, а `git remote origin` настроен.

**Проблема:** Конфликт портов
**Решение:** Порты наружу открыты только для: 3002 (gateway), 8080 (frontend), 3001 (grafana), 9090 (prometheus). Остальные — внутри Docker-сети.

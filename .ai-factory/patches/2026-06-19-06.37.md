# In-method import and short variable names in business logic

**Date:** 2026-06-19 06:37
**Files:**
- services/vk-service/app/services/ingestion/collector.py
- services/vk-service/app/services/ingestion/pipeline.py
**Severity:** low

## Problem

`DataCollector.collect()` contained `import logging` at the top of the method body, and `IngestionPipeline.execute()` re-instantiated `logger = logging.getLogger(...)` inside the `except` block. Both patterns cause the logger to be re-created on every call instead of once at import time.

Additionally, `collector.py` used single-letter and abbreviated variable names in business logic (`p`, `g`, `uid`, `idx`, `exc`), making the code harder to read without surrounding context.

`IngestionPipeline` duplicated `task_run.status = ...` / `task_run.finished_at = ...` mutation blocks in both the success path and the failure path, and the `is_infra_error` classification logic was inlined inside the `except` block as a 10-line if/elif chain.

## Root Cause

- `import logging` inside a method is a Python anti-pattern inherited from an early iteration of the module where logging was added quickly without promoting it to module scope.
- Short variable names (`p`, `g`) matched the VK API response field names (`profiles`, `groups`) and were left as-is during incremental growth of the method.
- The task_run mutation block was copy-pasted when the failure path was added, without extracting a shared helper.
- `is_infra_error` logic grew inline without a dedicated method, making `execute()` read as one monolithic block.

## Solution

**collector.py:**
- Added `import logging` and `logger = logging.getLogger("vk-service.ingestion")` at module level.
- Renamed: `p` Ôćĺ `profile`, `g` Ôćĺ `group_profile`, `uid` Ôćĺ `user_id`, `idx` Ôćĺ `post_index`, `exc` Ôćĺ `error`.
- Renamed `_enrich_user_profiles` local: `user_ids` Ôćĺ `user_ids_without_photo`, `enriched` Ôćĺ `enriched_users`.
- Renamed `_author_payload` local: `vk_id` Ôćĺ `author_vk_id`, `name` Ôćĺ `display_name`.

**pipeline.py:**
- Moved `logger` to module level (removed re-instantiation from `except` block).
- Extracted `_mark_task_done(task_run, result)` ÔÇö sets status/timestamps/counts for success.
- Extracted `_mark_task_failed(task_run, error)` ÔÇö sets status/timestamps/counts for failure.
- Extracted `@staticmethod _is_infrastructure_error(exc)` ÔÇö returns bool, replaces inline chain.
- Extracted `_handle_fail_callback_conflict(callback_exc, task_run, sanitized_error, original_exc)` ÔÇö handles 409 vs re-raise logic for the fail callback.

No behaviour change. All 48 tests pass.

## Prevention

- **Module-level loggers only.** `import logging` and `getLogger(...)` must be at the top of the file, never inside functions or methods. Add to code review checklist.
- **Full names in business logic.** Single-letter names are only acceptable for counters (`i`, `j`) and very short lambdas. Variables that represent domain objects (`profile`, `group`, `user_id`) must use their full name.
- **Extract duplicated mutation blocks immediately.** Whenever the same set of field assignments appears in two branches of the same method, extract a helper before the PR is merged.
- **Classify errors in a dedicated method.** `is_<condition>` logic with isinstance chains belongs in a named method or static method, not inline inside try/except.

# TaskEventsService and IngestionService: SRP and DI cleanup

**Date:** 2026-06-19 06:28
**Files:**
- services/vk-service/app/domain/events/task_events.py (new)
- services/vk-service/app/domain/events/task_event_mapper.py (new)
- services/vk-service/app/services/task_events_service.py
- services/vk-service/app/services/ingestion_service.py
- services/vk-service/app/bootstrap.py
- services/vk-service/app/tasks/kafka_consumer.py
- services/vk-service/tests/test_task_consumer.py
**Severity:** medium

## Problem

Two services violated clean architecture principles:

1. **TaskEventsService** — `TaskEvent` Pydantic model and its domain-parsing methods (`task_id()`, `owner_user_id()`, `scope()`, `mode()`, `group_ids()`, `post_limit()`) lived in the same file as the service class, violating SRP. Domain logic should not live inside a Pydantic model.

2. **IngestionService** — `DataCollector` and `IngestionPipeline` were instantiated *inside* the service constructor using a `svc = self` workaround and lambda closures (`lambda msg: svc._sanitize_error(msg)`). This is a DI violation: the service controlled object creation instead of receiving pre-wired collaborators.

## Root Cause

- `TaskEvent` payload accessors grew organically as conveniences without enforcing the boundary that Pydantic models should be plain data holders.
- `IngestionService` originally had no need for DI of inner objects but accumulated logic that required `_sanitize_error` to be available at construction time, leading to the closure workaround.

## Solution

**TaskEventsService:**
- Moved `TaskEvent` (pure Pydantic model, no methods) to `app/domain/events/task_events.py`.
- Created `app/domain/events/task_event_mapper.py` with a static `TaskEventMapper` class containing all payload extraction methods (`get_task_id`, `get_owner_user_id`, `get_scope`, `get_mode`, `get_group_ids`, `get_post_limit`).
- Updated `TaskEventsService` to import from domain and call `TaskEventMapper.*` methods.
- Extracted conflict handling into `_handle_conflict` and `_extract_conflict_detail` private methods.
- Updated `kafka_consumer.py` and all tests to import `TaskEvent` from the new domain path.

**IngestionService:**
- Added optional `collector: DataCollector | None` and `pipeline: IngestionPipeline | None` constructor parameters.
- When provided, they are used directly (DI path from `bootstrap.py`).
- When absent, default construction happens inside the service (backwards-compatible with tests).
- Removed `svc = self` workaround — `_sanitize_error` is now passed as a bound method directly (`on_error=self._sanitize_error`).
- In `bootstrap.py`, `get_ingestion_service` now explicitly constructs `sanitize_error`, `DataCollector`, and `IngestionPipeline` before injecting them into `IngestionService`.

## Prevention

- Pydantic models should be plain data holders — no domain logic, no business methods. If payload extraction is needed, put it in a dedicated mapper/adapter class.
- Services should receive their collaborators pre-wired from outside (factory/bootstrap), not construct them internally.
- Closures referencing `self` inside `__init__` are a design smell — restructure to pass bound methods or extract a factory.

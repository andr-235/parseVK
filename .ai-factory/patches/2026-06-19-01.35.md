# Missing `keyword_source` parameter in CommentsGatewayService

**Date:** 2026-06-19 01:35
**Files:** services/api-gateway/app/modules/comments/service.py, services/api-gateway/tests/test_comments_service.py
**Severity:** high

## Problem

500 Internal Server Error on `GET /api/v1/comments?offset=0&limit=25`.
Error: `TypeError: CommentsGatewayService.get_comments() got an unexpected keyword argument 'keyword_source'`.
The comments page was completely blocked.

## Root Cause

`router.py` passed `keyword_source=keywordSource` to `service.get_comments()` and `service.get_comments_cursor()`, but neither method declared this parameter. The parameter was added to the router (with the query param definition `LIST_KEYWORD_SOURCE_QUERY`) but forgotten in the service layer.

Additionally, `keywords` was typed as `str | None` in the service while the router typed it as `list[str] | None` — a latent type mismatch that would cause incorrect serialization for multi-keyword queries.

The router→service contract was broken because there was no type-level enforcement between the two layers. The fake gateway service in router tests used `**kwargs`, which silently absorbed the mismatch and prevented the test from catching it.

## Solution

1. Added `keyword_source: str | None = None` parameter to `get_comments()` and `get_comments_cursor()` in `CommentsGatewayService`
2. Forwarded the value as `"keywordSource"` in the params dict to the moderation backend
3. Changed `keywords` type from `str | None` to `list[str] | None` to match the router
4. Added unit test `test_get_comments_passes_keyword_source` that verifies the param is forwarded correctly

## Prevention

- When adding a new query parameter to a gateway router, always add it to the corresponding service method simultaneously
- Gateway service tests should use a recording client that checks exact params, not `**kwargs`
- Add a type-level pattern (e.g., TypedDict for params) shared between router and service to prevent similar drift

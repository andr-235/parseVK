# Переименование сервисов в vk-service для архитектурной ясности

**Date:** 2026-06-19 05:41
**Files:**

- services/vk-service/tests/test_vk_outbox.py
**Severity:** low

## Problem

Имена трех сервисов в директории `app/services/` вводили в заблуждение или были недостаточно точными относительно их архитектурной роли:
1. `vk_api_service.py` (класс `VkApiService`) звучал как клиент к API VK, хотя он является Application Service для управления группами.
2. `task_handler.py` (класс `TaskEventsHandler`) имел неочевидное имя для сервиса обработки доменных событий.
3. `outbox_service.py` (класс `OutboxService`) опционально переименован для улучшения структуры.

## Root Cause

Исторически сложившиеся имена файлов при переносе логики из старой Three-Tier структуры модулей не были до конца выровнены с терминами Layered Architecture и их реальной ответственностью.

## Solution

1. Переименованы файлы и соответствующие классы:
   - `vk_api_service.py` -> `vk_groups_service.py` (класс `VkGroupsService`).
   - `task_handler.py` -> `task_events_service.py` (класс `TaskEventsService`).
   - `outbox_service.py` -> `domain_events_service.py` (класс `OutboxService` сохранен).
2. Обновлены все импорты и вызовы зависимостей в:
   - `app/api/dependencies.py` (зависимость переименована в `get_vk_groups_service_dep`).
   - `app/api/routers/vk_api.py`
   - `app/bootstrap.py`
   - `app/tasks/kafka_consumer.py`
   - `tests/test_task_consumer.py`
   - `tests/test_vk_outbox.py`
3. Успешно запущены тесты для подтверждения работоспособности.

## Prevention

- При проектировании слоев Layered Architecture давать файлам и классам имена, отражающие их роль в бизнес-логике (например, `xxx_service.py` и `XxxService`), избегая пересечений с названиями клиентов (`xxx_client.py`).
- Отражать архитектурные типы сервисов в их именах (Domain Service -> `domain_events_service`, Application Service -> `task_events_service`).

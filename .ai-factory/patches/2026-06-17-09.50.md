# Isolate Service Dependency Injection from Router Modules

**Date:** 2026-06-17 09:50
**Files:** 
- `services/content-service/app/modules/monitoring/dependencies.py`
- `services/content-service/app/modules/monitoring/router.py`
- `services/content-service/app/modules/listings/dependencies.py`
- `services/content-service/app/modules/listings/router.py`
- `services/content-service/app/modules/telegram_tgmbase/dependencies.py`
- `services/content-service/app/modules/telegram_tgmbase/router.py`
- `services/content-service/tests/test_content_api.py`
- `services/content-service/tests/test_monitoring.py`
- `services/content-service/tests/test_listings_api.py`
- `services/content-service/tests/test_dependencies.py`
**Severity:** medium

## Problem

A test collection failure `ImportError: cannot import name 'get_content_service' from 'app.modules.content.router'` occurred in `tests/test_content_api.py`. This happened because the router module was refactored and the dependency provider `get_content_service` was cleanly isolated to `dependencies.py`, but the tests were still importing it from the router file. Additionally, other feature modules (`monitoring`, `listings`, `telegram_tgmbase`) were defining dependency/service getters directly in their router files, violating the three-tier layer architecture.

## Root Cause

- **Wrong Assumption / Tight Coupling:** Test suites and HTTP router modules assumed service factories and repository initializations belonged inside router modules.
- **Layer Violation:** Having database session resolution and repository/service instantiations inside router files violates the strict three-tier architecture axiom: *"Never import service factories or business logic from router modules — routers are HTTP-only layer."* When a router layer is refactored, it breaks other layers and test overrides that tightly coupled to the router file.

## Solution

1. **Dependency Separation:** Extracted service-instantiating dependency functions (`get_monitoring_service`, `get_listings_service`, `get_tgmbase_service`) out of router files into distinct, clean `dependencies.py` modules within their respective feature folders.
2. **Clean Imports:** Updated routers and corresponding unit/integration test suites to import the dependency resolvers from `dependencies.py` instead of the HTTP routers.
3. **Traceability:** Added diagnostic logging using the `[FIX]` prefix to all service dependencies to track resolution and lifecycle in debug/test envs.
4. **Validation Test:** Added `tests/test_dependencies.py` to assert that all four service dependencies successfully resolve to their expected clean architectural classes without router side-effects.

## Prevention

- **Architectural Check:** During code review, verify that routers contain only FastAPI endpoint definitions (`@router.get`, etc.) and basic HTTP validations, while database dependency resolution (`get_session`) and service building logic reside in a `dependencies.py` file.
- **Test Integrity:** Never import dependency providers from router files in test suites; always import from `dependencies.py` or `dependencies` modules.

### Subject language

### Subject format
**Source**: AGENTS.md (lines 162-163)
**Rule**: Subject MUST follow format `<type>(<scope>): <short summary>`. Subject MUST NOT exceed 72 characters. Do NOT capitalize the first letter after the colon. Do NOT end with a period.

### No task ID in subject
**Source**: AGENTS.md (line 176)
**Rule**: Do NOT place task IDs (e.g., FASTAPI-MIG-001) in the commit subject. Task IDs and issue references go in the commit body as `Refs: <id>` or `Closes #<number>`.

### Allowed types
**Source**: AGENTS.md (lines 168-173), base aif-commit SKILL.md
**Rule**: Allowed commit types and their usage in this project:
- `feat` — new feature
- `fix` — bug fix
- `docs` — documentation only
- `style` — code style (formatting, semicolons)
- `refactor` — code change that neither fixes a bug nor adds a feature
- `perf` — performance improvement
- `test` — adding or modifying tests
- `build` — build system or dependencies
- `ci` — CI configuration
- `chore` — maintenance tasks (deps, version bumps, annotations, etc.)

### Scope values

Use a generic scope only when the change spans multiple services.

### Branch naming convention
**Source**: AGENTS.md (lines 185-189)
**Rule**: Branch name format: `<type>/<issue-number>-<short-kebab-summary>`. For AI-generated branches: `ai/mbp-<issue-number>-<slug>` (generated by parsevkctl). Do NOT create branches from the default branch (`main`).

### Commit body convention
**Source**: AGENTS.md (lines 178-183)
**Rule**: When the commit relates to a GitHub issue or task, include references in the body:
- `Refs: <TASK-ID>` for task tracking references
- `Closes #<NUMBER>` when this commit closes an issue

### PR title and body format
**Source**: AGENTS.md (PR section)
**Rule**: PR title uses the same `<type>(<scope>): <short summary>` format. PR body MUST include:
- `Closes #ISSUE_NUMBER` line
- `## Summary` section with bullet points
- `## Validation` section with a checklist

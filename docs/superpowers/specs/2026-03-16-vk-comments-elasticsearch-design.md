# Дизайн: Elasticsearch для поиска по комментариям ВК

Дата: 2026-03-16

## Контекст

Сейчас страница комментариев работает в основном от PostgreSQL через `Comment`, `Post` и предвычисленные связи `CommentKeywordMatch`. Это хорошо закрывает сценарий "показать комментарии по точным ключевым словам", но не решает качественно поиск по похожим формулировкам, опечаткам и словоформам русского текста.

Нужен отдельный поисковый слой, который:

- ищет по русскому тексту комментария;
- учитывает текст поста как контекст;
- не ломает текущую keyword-based модель;
- обновляется почти сразу после записи данных в PostgreSQL;
- поддерживает два режима выдачи: `Комментарии` и `Посты`.

## Цели

- Добавить нечеткий поиск по комментариям ВК для русского языка.
- Поддержать поиск по словоформам и похожим формулировкам.
- Искать одновременно по тексту комментария и тексту поста.
- Оставить текущие `CommentKeywordMatch` как отдельный сигнал для фильтрации и буста.
- Сохранить `Комментарии` как режим выдачи по умолчанию и добавить переключатель `Посты`.

## Не входит в объём

- Полная замена PostgreSQL-выборок во всех остальных модулях.
- Мультиязычный поиск.
- Семантический/vector search.
- Автоматическое удаление текущего keyword-based режима.

## Подход

Рекомендованный вариант: отдельный Elasticsearch-индекс поверх read-модели комментариев, где один документ соответствует одному комментарию и содержит текст комментария плюс контекст поста.

Текущая PostgreSQL-модель остаётся источником истины. Elasticsearch используется как search read-model для fuzzy/full-text сценариев. Существующая логика `CommentKeywordMatch` не удаляется и применяется как:

- fallback-режим;
- фильтр;
- дополнительный сигнал релевантности.

## Архитектура

### Источники данных

- `Postgres` остаётся source of truth для `Comment`, `Post`, `Author`, `Group`, `CommentKeywordMatch`.
- `Elasticsearch` хранит денормализованный индекс для поиска.

### Слои backend

- `comments` и текущие сервисы записи продолжают сохранять данные в Postgres.
- Добавляется отдельный модуль, например `comments-search`, который отвечает за:
  - построение поискового документа;
  - индексирование и переиндексацию;
  - выполнение поисковых запросов;
  - маппинг ответов Elasticsearch в DTO страницы комментариев.

### Слои frontend

Страница комментариев получает:

- новый режим запроса через search API;
- переключатель режима выдачи `Комментарии / Посты`;
- сохранение `Комментарии` как режима по умолчанию;
- fallback на текущую keyword-based выборку при недоступности search-сервиса или feature flag.

## Модель документа в индексе

Один документ Elasticsearch соответствует одному комментарию.

Минимальные поля:

- `commentId`
- `postId`
- `ownerId`
- `vkCommentId`
- `authorVkId`
- `groupId`
- `publishedAt`
- `source`
- `isRead`
- `commentText`
- `postText`
- `keywordIds`
- `keywordWords`
- `authorName`
- `groupName`

### Принципы моделирования

- `commentText` и `postText` индексируются как отдельные поля.
- Совпадения в `commentText` ранжируются выше, чем в `postText`.
- `keywordWords` не заменяют full-text поиск, а используются для фильтров и boost.
- Документ не должен дублировать лишние тяжёлые поля, не нужные в выдаче.

## Поисковое поведение

### Язык

Поиск настраивается только для русского языка.

Ожидаемое поведение:

- обработка словоформ русского языка;
- ограниченный fuzzy для опечаток;
- релевантность выше при совпадении в комментарии;
- релевантность ниже при совпадении только в тексте поста.

### Поисковые поля

Запрос ищет по:

- `commentText`
- `postText`

Рекомендуемые веса:

- `commentText`: основной вес;
- `postText`: пониженный вес;
- `keywordWords`: только как дополнительный boost;
- опционально `authorName` и `groupName`: низкий вес, если это окажется полезно для операторов.

### Ранжирование

Выше ранжируются документы, где:

- совпадение найдено в `commentText`;
- совпадение найдено и в `commentText`, и в `postText`;
- у комментария есть `keywordMatches`, совпадающие с текущими фильтрами;
- комментарий более свежий, если нужен слабый freshness boost.

### Ограничения fuzzy

Чтобы не размывать выдачу:

- для коротких запросов fuzzy должен быть ограниченным;
- для длинных запросов можно давать более мягкий matching;
- keyword-based фильтры не должны зависеть от fuzzy и должны работать детерминированно.

## Режимы выдачи

### Комментарии

Режим по умолчанию.

Один hit = один комментарий.

В выдаче показываются:

- текст комментария с подсветкой;
- краткий контекст поста;
- автор;
- группа;
- дата;
- совпавшие keywords, если есть.

### Посты

Режим группировки по посту.

Один элемент выдачи = пост, внутри которого перечислены подходящие комментарии.

Реализация может строиться через:

- collapse/grouping по `postId`; или
- агрегацию по посту с отдельной подвыборкой комментариев.

Главное требование: поиск остаётся по comment-documents, а не через отдельный индекс постов.

## Синхронизация индекса

### Near-real-time обновление

Новые и изменённые данные должны попадать в индекс почти сразу после записи в PostgreSQL.

Правила:

- при создании комментария индексируется новый comment-document;
- при изменении текста комментария документ переиндексируется;
- при изменении текста поста переиндексируются все комментарии этого поста;
- при удалении или логическом скрытии комментария документ удаляется из индекса или помечается флагом исключения из поиска.

### Надёжность записи

Запись в Postgres не должна зависеть от доступности Elasticsearch.

Если индекс временно недоступен:

- сохранение данных в БД всё равно проходит;
- событие индексирования помещается в retry/outbox-механику;
- отдельный reindex job восстанавливает консистентность.

## Отказоустойчивость

- Если Elasticsearch недоступен, UI не должен падать.
- Нужен контролируемый fallback: либо возврат к текущему keyword-based режиму, либо понятная ошибка с выключением fuzzy-поиска.
- Нужен фоновый bulk reindex для:
  - первичной загрузки индекса;
  - восстановления после сбоев;
  - повторной индексации после изменения mapping/analyzer.

## API

Добавляется отдельный endpoint поиска комментариев, например:

`POST /comments/search`

Тело запроса:

- `query`
- `viewMode: "comments" | "posts"`
- `page`
- `pageSize`
- существующие фильтры, где применимо:
  - `keywords`
  - `keywordSource`
  - `readStatus`
  - возможно диапазон дат, автор, группа

Ответ должен содержать:

- список результатов в выбранном режиме;
- общее количество;
- данные для пагинации;
- подсветку совпадений;
- признак, что выдача пришла из search-режима, а не из fallback.

## Feature rollout

Рекомендован безопасный rollout:

1. Не заменять сразу текущую страницу полностью.
2. Добавить новый режим поиска под feature flag.
3. Сохранить текущую keyword-based выборку как baseline и fallback.
4. Сравнить качество выдачи и устойчивость на реальных запросах операторов.

## Риски

- Новый инфраструктурный контур: Elasticsearch, mapping, мониторинг, резервное восстановление.
- Усложнение near-real-time синхронизации.
- Массовая переиндексация комментариев при изменении текста поста.
- Качество русскоязычного анализа зависит от настройки analyzer и требует проверки на реальных запросах команды.

## Почему не только Postgres

Postgres можно усилить через `pg_trgm`, `tsvector` и GIN, но для целевого сценария он хуже подходит как основной инструмент, потому что:

- fuzzy и словоформы будут менее гибкими;
- релевантность и отладка скоринга сложнее;
- дальнейшее расширение в сторону аналитического поиска и фасетов будет дороже поддерживать.

Поэтому предпочтителен Elasticsearch как отдельный search read-model, а PostgreSQL должен остаться основным хранилищем и fallback-контуром.

## Критерии успеха

- Оператор находит русские комментарии по похожим формулировкам, а не только по точным keyword matches.
- Совпадения в тексте комментария стабильно ранжируются выше совпадений только в тексте поста.
- Режим `Комментарии` остаётся основным и понятным.
- Режим `Посты` даёт обзор контекста без отдельного индекса постов.
- Запись данных не ломается при временной недоступности Elasticsearch.
- Текущий keyword-based режим остаётся доступным как fallback.

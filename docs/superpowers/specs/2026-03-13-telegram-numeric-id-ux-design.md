# Дизайн: честный UX для unknown numeric Telegram ID

## Контекст

После внедрения `TelegramIdentifierResolverService` сервис Telegram больше не падает на `CHANNEL_INVALID`, но для запроса вида:

- `-1001157519810`

возвращает контролируемую ошибку:

- `Cannot resolve Telegram chat by numeric ID without saved metadata`

Это соответствует текущей архитектуре:

- numeric ID работает только если чат уже известен системе;
- у нас уже сохранены `username` и/или `accessHash`;
- по одному голому `-100...` первый резолв не делается.

С технической точки зрения это поведение корректно. Проблема в UX:

- пользователь воспринимает ответ как "сервис не работает";
- текст ошибки не объясняет, что именно надо сделать для первого импорта.

## Ключевое ограничение

Если у пользователя есть только `-100...` и больше ничего:

- нет `@username`;
- нет `t.me/...`;
- нет invite-ссылки;
- нет уже сохранённого `accessHash`;

то Telegram API не даёт надёжно открыть такой чат "с нуля".

Значит текущая задача не в том, чтобы "починить магический first-resolve", а в том, чтобы:

- честно объяснить ограничение;
- подсказать корректный первый шаг;
- сохранить удобный последующий сценарий по numeric ID после bootstrap.

## Цель

Сделать поведение Telegram sync понятным для пользователя, когда он вводит неизвестный numeric chat id.

После изменения система должна:

- явно отличать known numeric ID от unknown numeric ID;
- отдавать actionable error message;
- подсказывать в UI, как выполнить первый импорт;
- после первого успешного sync по `@username`/ссылке позволять повторный sync по numeric ID.

## Не цели

- Не реализовывать "магическое" открытие канала по одному `-100...`.
- Не добавлять сложный внешний enrichment pipeline.
- Не добавлять автоматический join по invite.

## Требования

### Функциональные

- Backend должен возвращать понятную доменную ошибку для unknown `-100...`.
- Frontend должен показывать пользователю конкретную инструкцию, а не абстрактный fail.
- UI должен явно сообщать, что numeric ID подходит только для уже известных чатов.
- Поведение known numeric ID не должно ломаться.

### Нефункциональные

- Ошибка должна быть краткой и однозначной.
- Пользователь должен понимать next step без чтения логов сервера.
- Изменения должны быть минимальными и не ломать текущую архитектуру resolver-а.

## Подходы

### Вариант 1. Честный two-step flow

Система не обещает first-resolve по голому `-100...`, а явно просит bootstrap через:

- `@username`
- публичную ссылку
- invite-ссылку

После первого успешного sync numeric ID работает как shorthand.

Плюсы:

- соответствует ограничениям Telegram API;
- минимально меняет архитектуру;
- снижает ложные ожидания пользователя.

Минусы:

- не решает кейс, когда у пользователя вообще нет ничего кроме numeric ID.

### Вариант 2. Пытаться "угадать" чат по дополнительным GramJS вызовам

Плюсы:

- иногда может сработать, если сущность уже была в сессии Telegram.

Минусы:

- для truly unknown numeric ID ненадёжно;
- создаёт ощущение "иногда работает, иногда нет";
- плохо объясняется пользователю.

### Вариант 3. Внешний metadata bootstrap

Использовать внешний источник сопоставления:

- `numeric ID -> username/accessHash`

Плюсы:

- реально решает кейс pure numeric ID, если внешний источник есть.

Минусы:

- это уже отдельная подсистема;
- не часть минимального fix.

### Решение

Выбрать вариант 1 как минимальный и честный.

## Архитектура

### Backend

`TelegramIdentifierResolverService` сохраняет текущую стратегию:

- public identifiers резолвятся сразу;
- numeric ID резолвится только через уже сохранённую metadata.

Изменение только в доменной формулировке ошибки.

Текущий текст:

- `Cannot resolve Telegram chat by numeric ID without saved metadata`

Новый текст должен объяснять действие:

- для первого импорта этого чата используйте `@username`, публичную ссылку или invite-ссылку;
- после первого успешного sync numeric ID можно использовать повторно.

### Frontend

Frontend должен перестать показывать этот кейс как "непонятную ошибку синхронизации".

Нужно:

- добавить helper text под полем identifier;
- показывать доменную ошибку как пользовательскую инструкцию;
- не маскировать её generic toast-ом.

Пример helper text:

- `Для первого импорта используйте @username, публичную или invite-ссылку. Numeric ID работает для уже известных чатов.`

## API/ошибки

### Отличать сценарии

Минимум три пользовательских сценария:

1. Unsupported format
2. Invite requires explicit join flow
3. Unknown numeric ID without metadata

Третий сценарий должен иметь особое сообщение:

- `Нельзя впервые импортировать чат только по numeric ID. Используйте @username, публичную или invite-ссылку. После первого sync numeric ID будет работать.`

Можно оставить это как обычный `400 Bad Request`, но с качественным message.

## UI-поведение

### Поле идентификатора

Под полем ввода стоит дать короткую постоянную подсказку:

- поддерживаются `@username`, `t.me/...`, invite links;
- numeric ID подходит только для уже известных чатов.

### Ошибка sync

При unknown numeric ID:

- не показывать только generic toast;
- выводить actionable текст рядом с формой или в основном error toast.

Минимальный scope:

- оставить toast;
- улучшить сообщение;
- добавить helper text в карточку sync.

## Тестирование

### Backend

- unit: unknown numeric ID возвращает точную actionable ошибку;
- unit: known numeric ID с metadata продолжает работать;
- unit: public identifier path не ломается.

### Frontend

- unit/component test на hook или карточку:
  - сервер вернул unknown numeric ID error;
  - пользователь видит понятную инструкцию.

### Manual

1. Ввести неизвестный `-100...`
   Expected: понятная инструкция для первого импорта.
2. Ввести `@username` того же чата
   Expected: sync проходит успешно.
3. Повторно ввести `-100...`
   Expected: sync проходит успешно по сохранённой metadata.

## Риски

- Пользователь может быть недоволен тем, что pure numeric ID не поддерживается "с нуля".
- Если helper text будет слишком длинным, UI станет шумным.
- Если фронт продолжит перекрывать доменную ошибку generic toast-ом, смысл backend-улучшения потеряется.

## Критерии успеха

- Пользователь понимает, почему unknown numeric ID не работает при первом импорте.
- Пользователь видит конкретный следующий шаг.
- После bootstrap через public identifier numeric ID работает как ожидается.
- Больше нет ощущения, что Telegram sync "сломался", когда на самом деле это ограничение API.

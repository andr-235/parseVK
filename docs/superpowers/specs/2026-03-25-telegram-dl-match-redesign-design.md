# Telegram DL Match Redesign Design

**Контекст**

Первая итерация `Telegram DL Match` уже добавила backend-модуль матчинга `DlContact -> tgmbase.user`, новые таблицы `dl_match_run` и `dl_match_result`, а также UI на странице `Выгрузка с ДЛ`. После выкладки проявились две проблемы:

- в deploy-сценарии не применяется новая tgmbase-миграция, из-за чего backend падает с ошибкой `The table public.dl_match_run does not exist in the current database`
- текущий UI страницы выглядит как длинная перегруженная портянка и не воспринимается как production-ready рабочий экран

Пользователь явно хочет не косметический фикс, а нормальный рабочий UX:
- разделить импорт DL и матчинг DL на две вкладки
- сделать плотный table-first интерфейс
- убрать ощущение длинной лендинговой страницы
- не забыть про backend-ошибку и исправить deploy

**Цель**

Сделать `Telegram DL` нормальным рабочим экраном из двух вкладок:
- `Импорт DL`
- `Матчинг DL`

Одновременно исправить deploy, чтобы новая схема `dl_match_run`/`dl_match_result` реально создавалась в target database.

## Рекомендованный подход

Рекомендуется единый redesign в рамках существующей страницы `Выгрузка с ДЛ`, но с разделением на две вкладки и обязательным deploy-fix.

Рассматривались альтернативы:
- оставить одну страницу и просто улучшить spacing
- вынести матчинг в отдельный route

Оставить всё на одной длинной странице нельзя: это не решает главную UX-проблему. Отдельный route тоже не нужен в первой версии, потому что импорт и матчинг логически относятся к одному разделу и должны оставаться рядом.

Лучший вариант:
- одна страница
- две вкладки
- компактный верхний блок
- явный рабочий таб `Матчинг DL`

## Обязательный backend-fix

### Проблема

Сейчас backend делает запросы к `dl_match_run`, но deploy-скрипт применяет только старую tgmbase SQL-миграцию:
- `20260324160500_add_dl_import_tables.sql`

Новая миграция:
- `20260325120000_add_dl_match_tables.sql`

в entrypoint не вызывается, поэтому таблицы для матчинга не создаются.

### Решение

В `docker/backend-entrypoint.sh` нужно явно добавить применение новой SQL-миграции `20260325120000_add_dl_match_tables.sql` после миграции DL-импорта.

### Ожидаемый результат

После deploy:
- таблицы `dl_match_run` и `dl_match_result` существуют в `tgmbase`
- `GET /api/telegram/dl-match/runs` больше не падает на отсутствии таблицы
- UI вкладки матчинга может загружаться без backend exception

## Информационная архитектура страницы

Страница `Выгрузка с ДЛ` остаётся одной route, но делится на два явных режима:

- `Импорт DL`
- `Матчинг DL`

Переключатель вкладок должен быть виден сразу под hero-блоком.

### Hero

Hero остаётся, но становится компактнее:
- короткий заголовок
- короткое пояснение
- без лишней высоты
- без длинных декоративных пустот

Hero не должен отталкивать рабочий контент далеко вниз.

## Вкладка `Импорт DL`

Эта вкладка сохраняет текущую бизнес-логику:
- загрузка файлов
- очередь/статусы
- история импортов

Но UI должен быть компактным и утилитарным. Это не аналитический экран, а операционный экран загрузки.

### Компоновка

Рекомендуемая раскладка:
- слева блок загрузки
- справа история импортов

Вкладка не должна содержать элементов матчинга.

## Вкладка `Матчинг DL`

Это основной redesign.

### Стартовое состояние

При открытии вкладки пользователь сразу видит:
- action bar
- фильтры
- плотную таблицу полной DL-базы

Не нужно начинать с карточек, длинных описаний или результатов по умолчанию.

### Основной сценарий

Пользователь:
1. Открывает вкладку `Матчинг DL`
2. Видит полную таблицу DL-контактов
3. При необходимости фильтрует базу
4. Нажимает `Найти совпадения`
5. После завершения экран переключается в режим результатов последнего запуска
6. При необходимости экспортирует `xlsx`
7. Одной кнопкой возвращается к полной DL-базе

## UX таба `Матчинг DL`

### Верхняя панель действий

Содержит:
- индикатор текущего режима: `Вся DL-база` или `Результаты последнего запуска`
- кнопку `Найти совпадения в tgmbase`
- кнопку `Показать всю DL-базу`
- кнопку `Выгрузить XLSX`

Правила:
- `Выгрузить XLSX` доступна только для завершённого последнего запуска
- `Найти совпадения` уходит в loading-state во время выполнения
- повторный запуск во время активного расчёта недоступен

### Summary последнего запуска

Если есть завершённый запуск, под action bar показывается компактная горизонтальная summary-лента:
- проверено
- совпадений
- ID
- username
- phone

Summary должна быть компактной. Это не набор больших hero-card, а короткая строка рабочих счётчиков.

### Основная таблица: режим полной базы

Таблица должна быть плотной и ориентированной на реальную работу с большим количеством строк.

Фильтры сверху:
- файл
- `telegramId`
- `username`
- `phone`

Колонки:
- файл
- ФИО
- `telegramId`
- `username`
- `phone`
- регион
- строка источника

Это рабочая таблица, не карточная витрина.

### Основная таблица: режим результатов

После запуска матчинга та же рабочая область переключается на таблицу найденных совпадений последнего запуска.

Колонки:
- DL ФИО
- DL `telegramId`
- DL `username`
- DL `phone`
- tgmbase `user_id`
- tgmbase `username`
- tgmbase `phone`
- тип совпадения
- файл

Над таблицей должны быть быстрые фильтры:
- `Все`
- `ID`
- `Username`
- `Phone`

Тип совпадения отображается компактными цветными badge, а не текстовой простынёй.

## Визуальное направление

Новый экран должен уйти от ощущения длинного тёмного лендинга.

### Принципы

- меньше пустой вертикали
- чёткая иерархия: вкладки -> action bar -> summary -> таблица
- table-first layout
- меньше крупных декоративных карточек
- плотные блоки с хорошим контрастом
- интерфейс должен выглядеть как аналитический инструмент, а не как маркетинговая страница

### Что убрать

- длинный поток секций сверху вниз
- ощущение, что матчинг является «ещё одним блоком ниже»
- лишние большие отступы между рабочими зонами
- чрезмерно декоративные крупные панели там, где нужна рабочая плотность

## Технический дизайн

### Backend

Нужно изменить:
- `docker/backend-entrypoint.sh`

Нужно проверить:
- корректный порядок применения tgmbase SQL-миграций
- повторный запуск контейнера не должен ломаться на `IF NOT EXISTS` миграции

### Frontend

Нужно переразложить модуль `telegram-dl-upload` вокруг двух режимов:
- `import`
- `match`

Ожидаемые единицы декомпозиции:
- контейнер страницы
- переключатель вкладок
- workspace для вкладки импорта
- workspace для вкладки матчинга
- action bar матчинга
- summary матчинга
- таблица полной базы
- таблица результатов

Текущая реализация `TelegramDlMatchWorkspace` должна быть упрощена и переработана в более строгий table-first экран.

## Обработка ошибок

### Backend/deploy

Нужно явно закрыть сценарий ошибки:
- если новая tgmbase-миграция не применена, backend не должен незаметно уйти в состояние, где UI ломается на первом запросе

Минимум для первой версии:
- deploy-скрипт реально применяет новую миграцию
- ручная проверка подтверждает создание таблиц

### Frontend

Для вкладки матчинга нужны ясные состояния:
- ошибка загрузки полной DL-базы
- ошибка загрузки последнего запуска
- ошибка запуска матчинга
- отсутствие результатов последнего запуска

Эти состояния должны быть встроены в рабочий layout, а не ломать страницу.

## Границы первой версии redesign

В первую версию входят:
- deploy-fix для новой tgmbase-миграции
- две вкладки внутри страницы `Выгрузка с ДЛ`
- компактный hero
- разделение `Импорт DL` и `Матчинг DL`
- table-first redesign вкладки `Матчинг DL`
- компактный summary последнего запуска
- режим полной DL-базы по умолчанию
- переключение в результаты последнего запуска после матчинга

В первую версию не входят:
- история запусков матчинга
- отдельный route для матчинга
- master-detail панель по строке
- live progress через websocket
- сложная аналитика по файлам или регионам

## Тестирование

### Backend

Нужны проверки на:
- наличие вызова новой tgmbase SQL-миграции в `backend-entrypoint.sh`
- корректность работы API матчинга после фиксированного deploy-контракта

### Frontend

Нужны проверки на:
- переключение вкладок `Импорт DL` / `Матчинг DL`
- стартовое открытие полной DL-базы во вкладке матчинга
- запуск матчинга и переход в таблицу результатов
- возврат к полной DL-базе
- корректную доступность кнопки `Выгрузить XLSX`

## Итог

Эта итерация не про добавление новой бизнес-логики, а про доведение фичи до рабочего состояния. Нужно одновременно закрыть инфраструктурную ошибку deploy и переработать UX вокруг двух вкладок, чтобы `Telegram DL` стал нормальным production-экраном, а не длинной неудобной страницей с поломанным backend path.

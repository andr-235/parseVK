# Дизайн: перепроверка групп VK

## Контекст

Сейчас задача парсинга групп VK работает в одном режиме: для выбранных групп берутся последние `N` постов, где `N` задаётся через `postLimit` и по умолчанию равно `10`. Затем для каждого поста загружаются комментарии, авторы и данные сохраняются в БД через существующий pipeline задач.

Нужно добавить отдельную пользовательскую фичу `Перепроверить группу` в разделе задач. Пользователь выбирает одну или несколько групп, после чего система проходит по всем постам каждой выбранной группы, которые доступны через текущий VK API без дополнительных обходных механизмов, загружает комментарии и сохраняет данные в БД без очистки уже существующих записей.

## Цели

- Добавить отдельный режим задачи `Перепроверить группу`.
- Сохранить текущий режим парсинга последних постов без изменений.
- Использовать существующий pipeline сохранения постов, комментариев и авторов.
- Сделать режим явно видимым в API, UI, websocket-обновлениях и истории задач.

## Не цели

- Не добавлять архивные или нештатные способы получения старых постов.
- Не удалять старые посты и комментарии перед перепроверкой.
- Не менять существующую модель выбора групп.
- Не перестраивать полностью очередь задач или CQRS-поток.

## Подходы

### Вариант 1. Явный режим задачи

Добавить новое поле `mode: 'recent_posts' | 'recheck_group'` в DTO, очередь, описание задачи и фронтенд-модели.

Плюсы:
- Чёткая семантика задачи.
- Понятное отображение в UI и истории.
- Минимум скрытой логики.

Минусы:
- Нужно протянуть новое поле через backend и frontend.

### Вариант 2. Использовать `postLimit = null` как сигнал полного прохода

Плюсы:
- Меньше новых полей.

Минусы:
- Неочевидная семантика.
- Путает обычный режим и перепроверку.
- Усложняет отображение задачи в UI.

### Вариант 3. Новый endpoint без отдельного режима в общей модели

Плюсы:
- Быстро изолируется на уровне контроллера.

Минусы:
- Провоцирует дублирование логики создания и исполнения задач.
- Хуже масштабируется для будущих режимов.

## Выбранное решение

Использовать вариант 1: добавить явный `mode`.

Базовый режим остаётся `recent_posts` и сохраняет текущее поведение с `postLimit`.
Новый режим `recheck_group` игнорирует лимит постов и запускает постраничный обход всей стены группы в рамках текущих ограничений VK API.

## Архитектура

### Backend

Нужно расширить контракт создания задач:

- `CreateParsingTaskDto`
- `CreateParsingTaskCommand`
- payload очереди
- сериализацию в `task.description`
- parser/mappers задачи

Новое поле:

```ts
type ParsingTaskMode = 'recent_posts' | 'recheck_group'
```

Поведение:

- `recent_posts`: использовать текущий сценарий с `postLimit`
- `recheck_group`: `postLimit` не используется, в description хранится `null`

Заголовок задачи должен зависеть не только от `scope`, но и от `mode`:

- `Парсинг группы: <name>`
- `Парсинг выбранных групп (N)`
- `Перепроверка группы: <name>`
- `Перепроверка групп (N)`

### Получение постов

Текущий `ProcessGroupHandler` всегда вызывает получение последних постов через ограничение `count`.

Нужно выделить два сценария:

- `recent_posts`: текущий вызов `getGroupRecentPosts({ ownerId, count: postLimit })`
- `recheck_group`: новый постраничный обход `wall.get` батчами, пока VK API возвращает элементы

Важно не накапливать все посты в памяти без необходимости. Полный проход должен обрабатываться по страницам или хотя бы последовательно по батчам.

### Сохранение данных

Существующий pipeline сохраняется:

- посты через `SavePostCommand`
- комментарии через `SaveCommentsCommand`
- авторы через `SaveAuthorsCommand`

Перепроверка не очищает данные. Она использует существующую upsert-логику и дописывает или обновляет записи.

## Поток данных

1. Пользователь в задачах выбирает действие `Перепроверить группу`.
2. Во фронтенде открывается текущий сценарий выбора групп, но для нового режима.
3. Фронтенд отправляет запрос на создание задачи с `mode = recheck_group`.
4. Backend создаёт запись задачи с `scope`, `groupIds`, `mode`, `postLimit: null`.
5. Очередь запускает задачу.
6. Для каждой группы обработчик получает все доступные посты батчами.
7. Для каждого поста сохраняется сам пост, затем подтягиваются комментарии и авторы.
8. Websocket и API возвращают режим задачи так, чтобы UI показывал корректное состояние.

## Ошибки и ограничения

Существующая обработка ошибок должна сохраниться:

- отключённая стена группы
- временные ошибки VK API
- отмена задачи
- ошибки отдельных групп с накоплением `failedGroups`

Дополнительно:

- режим `recheck_group` потенциально сильно дольше обычного режима
- в UI нужно явно показывать, что это перепроверка, а не задача с лимитом постов
- если у группы отключена стена, она пропускается так же, как и сейчас

Ограничение продукта:

- проход только по постам, доступным через текущий VK API без архивных обходов и специальных механизмов

## Изменения во frontend

Нужно добавить второй пользовательский сценарий создания задачи:

- либо отдельную кнопку `Перепроверить группу`
- либо отдельное действие в текущем модальном сценарии создания задачи

Требования к UI:

- пользователь выбирает несколько групп
- для режима перепроверки не запрашивается и не отображается `postLimit`
- в таблице задач и в деталях задачи отображается режим `Перепроверка группы`
- если задача создана в старом режиме, текущий UI с лимитом постов продолжает работать

## Совместимость

Обратная совместимость обязательна:

- если `mode` не передан, backend использует `recent_posts`
- существующие задачи без `mode` должны корректно отображаться как обычный режим
- автоматизация задач с `postLimit` должна сохранить текущее поведение

## Тестирование

### Backend

- тест на создание задачи в режиме `recent_posts` по умолчанию
- тест на создание задачи в режиме `recheck_group`
- тест на сериализацию и парсинг `mode` в `task.description`
- тест на выбор правильной стратегии получения постов в `ProcessGroupHandler`
- тест на постраничный обход всех постов группы до пустой страницы

### Frontend

- тест на создание задачи `Перепроверить группу` с корректным payload
- тест на отображение режима перепроверки в списке задач
- тест на отсутствие отображения числового лимита для режима перепроверки

### Регрессия

- тесты текущего сценария парсинга последних постов
- проверка, что pipeline сохранения постов, комментариев и авторов не меняется

## Риски

- полный проход по стене больших групп может работать заметно дольше стандартной задачи
- при неаккуратной реализации можно перегрузить память, если сначала загружать все посты, а потом обрабатывать
- нужно аккуратно сохранить совместимость старых задач, у которых в description ещё нет поля `mode`

## Итог

Добавляется новый явный режим задачи `recheck_group`, который использует существующую инфраструктуру задач и сохранения данных, но меняет источник списка постов: вместо последних `N` постов выполняется полный постраничный обход доступной стены группы через штатный VK API. Это даёт понятную продуктовую семантику, не ломает текущий режим парсинга и сохраняет совместимость с существующим UI и историей задач.

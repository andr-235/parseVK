# Дизайн: production-страница поиска по tgmbase

Дата: 2026-03-13

## Контекст

В проекте уже используется основная база `vk_api` и существующий Prisma-клиент. На сервере появилась вторая PostgreSQL база `tgmbase`. Нужна отдельная production-страница, на которой оператор может вставить одно или много значений и получить:

- найденные профили по `telegramId`, `username`, `phoneNumber`;
- список групп и чатов, где фигурирует пользователь;
- список пользователей, с кем было взаимодействие;
- тексты сообщений с контекстом взаимодействия.

Решение должно использовать Prisma и не смешивать модели `vk_api` и `tgmbase` в одном клиенте.

## Цели

- Дать оператору быстрый массовый поиск по `tgmbase`.
- Поддержать поиск по `telegramId`, `username`, `phoneNumber`.
- Показать результат в двух слоях: сводка по батчу и детальные карточки по каждому вводу.
- Поддержать работу с большими объёмами сообщений через пагинацию или ленивую догрузку.
- Сохранить типобезопасность и архитектурную изоляцию новой БД.

## Не входит в текущий объём

- Изменение существующих сценариев синка Telegram.
- Перенос логики в текущую страницу `Telegram`.
- Редактирование записей в `tgmbase`.
- Полнотекстовая поисковая система сверх трёх согласованных идентификаторов.

## Подход

Рекомендованный вариант: отдельная защищённая страница и отдельный backend-модуль, работающий с `tgmbase` через второй Prisma-клиент.

Почему выбран этот вариант:

- сохраняется единый подход проекта через Prisma;
- `vk_api` и `tgmbase` остаются изолированными;
- новый функционал не перегружает существующий Telegram-модуль;
- проще контролировать права доступа, лимиты и производительность.

## Архитектура

### Frontend

Добавляется отдельный route, например `/tgmbase-search` или `/telegram-intel`.

Страница состоит из трёх зон:

1. Hero + форма массового ввода.
2. Сводная таблица по всем введённым значениям.
3. Детальные карточки результатов по каждому значению.

### Backend

Добавляется отдельный NestJS-модуль поиска по `tgmbase`.

Состав:

- controller для HTTP endpoint;
- service с orchestration-логикой поиска;
- отдельный Prisma service для `tgmbase`;
- mapper/normalizer для нормализации запроса и построения DTO;
- при необходимости repository/query-layer для сложных выборок.

### Prisma

Для `tgmbase` создаётся отдельный Prisma schema, например `api/prisma/tgmbase.prisma`.

Предлагаемая схема:

- отдельный datasource с `TGMBASE_DATABASE_URL`;
- отдельный generator с output в `api/src/generated/tgmbase`;
- отдельный `TgmbasePrismaService`, не зависящий от основного Prisma-клиента `vk_api`.

Это позволяет:

- держать модели двух БД раздельно;
- сохранять type-safe доступ к данным;
- при необходимости использовать `queryRaw` локально внутри Prisma-based слоя, не ломая архитектуру.

## Пользовательский сценарий

Оператор открывает страницу, вставляет одно или несколько значений построчно, нажимает `Найти` и получает:

- сводную таблицу по всем входам;
- для каждого входа отдельный блок с деталями;
- в блоке доступны группы, контакты и сообщения;
- сообщения отдаются постранично или догружаются по мере запроса.

## UI/UX

### Верхняя часть страницы

- hero-блок с названием уровня `Telegram Intelligence`;
- короткое описание поддерживаемых идентификаторов;
- большая `textarea` для вставки списка;
- подсказки по формату ввода;
- кнопки `Найти` и `Очистить`.

### Сводная таблица

Таблица показывает по каждому входному значению:

- исходное значение;
- определённый тип (`telegramId`, `username`, `phoneNumber`, `invalid`);
- статус (`found`, `not_found`, `ambiguous`, `error`);
- краткую идентификацию найденного профиля;
- количество групп;
- количество контактов;
- количество сообщений.

Клик по строке таблицы раскрывает или прокручивает к соответствующей детальной карточке.

### Детальная карточка

Каждая карточка результата содержит:

- базовый профиль: имя, `telegramId`, `username`, `phoneNumber`, флаги;
- секцию `Группы`;
- секцию `Контакты`;
- секцию `Сообщения`.

В `Сообщениях` нужны фильтры:

- только личные;
- только групповые;
- по конкретной группе;
- по конкретному контакту.

Визуальное направление:

- production-экран аналитического типа;
- аккуратный тёмный интерфейс в языке текущего проекта;
- плотная, читаемая композиция без перегруза;
- рассчитан на большие объёмы данных.

## Контракт API

### Основной endpoint

`POST /tgmbase/search`

### Запрос

Тело запроса:

- `queries: string[]` — массив значений из формы;
- `messagePage` и `messagePageSize` — пагинация сообщений;
- опциональные фильтры по типу взаимодействия, группе и контакту.

### Ответ

Ответ включает:

- `summary` — агрегаты по всему батчу;
- `items[]` — результаты по каждому входному значению.

Каждый `item` содержит:

- `query`;
- `queryType`;
- `status`;
- `profile`;
- `groups`;
- `contacts`;
- `messagesPage`;
- `stats`.

### Статусы результата

- `found` — найден основной профиль;
- `not_found` — совпадения не найдены;
- `ambiguous` — найдено несколько кандидатов;
- `invalid` — вход не соответствует допустимому формату;
- `error` — ошибка обработки конкретного элемента.

## Правила поиска

### Поддерживаемые идентификаторы

Поиск выполняется по:

- `telegramId`;
- `username`;
- `phoneNumber`.

### Массовый режим

Выбран режим:

- сводная таблица сверху;
- детальный блок по каждому значению ниже.

### Что считать взаимодействием

Взаимодействием считаются:

- прямые сообщения между пользователями;
- сообщения в общих группах и чатах.

### Сообщения

Сообщения показываются не целиком одним массивом, а через пагинацию или ленивую догрузку, чтобы интерфейс оставался рабочим на больших объёмах данных.

## Обработка сложных случаев

- Один вход может давать несколько совпадений: возвращаем `ambiguous` и список кандидатов.
- Ошибка по одному элементу не валит весь батч.
- Пустой результат считается корректным ответом.
- Для батча вводится лимит, например до 200 строк за один запрос.

## Безопасность

- Доступ только для авторизованных пользователей; при необходимости можно ограничить администратором.
- Для `tgmbase` используется отдельный read-only пользователь БД.
- DTO не должны утягивать лишние чувствительные поля.
- В application-логи не записываются большие текстовые массивы сообщений.

## Производительность

- Основной запрос возвращает summary и первую страницу сообщений.
- Дальнейшие страницы сообщений подгружаются отдельно.
- На backend нужны ограничения по размеру батча и page size.
- Для часто используемых полей поиска нужны индексы в `tgmbase` либо проверка их наличия.

## Тестирование

### Backend

- unit-тесты на нормализацию входа;
- unit-тесты на определение типа идентификатора;
- unit-тесты на маппинг DTO;
- integration-тесты сервиса поиска с моками Prisma-клиента `tgmbase`.

### Frontend

- тест массового ввода;
- тест сводной таблицы;
- тест раскрытия детального результата;
- тест пагинации и фильтрации сообщений;
- тесты на `not_found`, `ambiguous`, `invalid`, частичный `error`.

## Риски

- Неизвестная схема `tgmbase` может потребовать уточнения связей и индексов.
- Для очень больших историй сообщений могут понадобиться специализированные SQL-запросы через Prisma.
- Если в `tgmbase` есть неоднозначные связи между пользователями и чатами, понадобится отдельная нормализация доменной модели.

## Критерии готовности

- Есть отдельная production-страница поиска.
- Backend читает `tgmbase` через отдельный Prisma-клиент.
- Поддержан поиск по `telegramId`, `username`, `phoneNumber`.
- Сверху есть сводная таблица, ниже детальные карточки.
- Видны группы, контакты и сообщения.
- Сообщения работают постранично.
- Добавлены релевантные тесты на frontend и backend.

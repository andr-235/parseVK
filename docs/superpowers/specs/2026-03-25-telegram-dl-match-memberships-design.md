# Telegram DL Match Memberships Design

**Контекст**

Сейчас результат DL-матчинга сохраняет в `tgmbaseUserSnapshot` только базовые поля найденного `user`: `user_id`, `username`, `phone`, `first_name`, `last_name`, флаги и `upd_date`. Пользователь хочет видеть ещё и все связи найденного контакта в `tgmbase`: в каких `group`, `supergroup` и `channel` он состоит.

Текущее ограничение в том, что в [tgmbase.prisma](/home/pc051/Project/parseVK/api/prisma/tgmbase.prisma) эти membership-связи не описаны. Значит реализация должна сначала опереться на реальную схему `tgmbase`, а уже затем встроить загрузку membership-данных в существующий батчевый pipeline DL-матчинга.

**Цель**

Расширить DL-матчинг так, чтобы для каждого найденного `tgmbase user` сохранялись и отдавались все его связи в `group`, `supergroup` и `channel` в формате `title + group_id`, а фронт и `xlsx`-экспорт показывали эти связи в результатах матчинга.

## Рекомендованный подход

Рекомендуется сохранять все связи прямо в `tgmbaseUserSnapshot`.

Рассматривались альтернативы:
- дозагружать связи на лету при `GET /results`
- хранить связи отдельной таблицей `dl_match_result_membership`

Обе альтернативы хуже для текущего проекта. On-demand lookup делает просмотр и экспорт тяжелее и убирает стабильность снимка на момент запуска. Отдельная таблица даст чище нормализованную модель, но для первой версии это лишние миграции и лишняя сложность без заметного выигрыша.

## Что считается связью

Для найденного `user` нужно вернуть все доступные membership-связи из `tgmbase`:
- `group`
- `supergroup`
- `channel`

Каждая связь в API и snapshot должна иметь единый формат:
- `type`
- `group_id`
- `title`

Пример:

```json
{
  "type": "group",
  "group_id": "123456789",
  "title": "ЖК Центральный чат"
}
```

`group_id` здесь выступает как унифицированный внешний идентификатор сущности. Для разных типов он будет заполняться из соответствующего поля таблицы:
- `group.group_id`
- `supergroup.supergroup_id`
- `channel.channel_id`

## Архитектура backend

### Основная идея

Во время обработки батча DL-матчинга после определения набора найденных `tgmbaseUserId` сервис должен одним bulk-шагом подтягивать все membership-связи для этих пользователей и приклеивать их в snapshot каждого matched user.

Pipeline становится таким:

1. Матчинг батча находит пользователей по `telegramId`, `username`, `phone`.
2. Из всех найденных совпадений собирается уникальный список `user_id`.
3. Отдельный lookup загружает все membership-связи этих пользователей.
4. Lookup группирует связи по `user_id`.
5. `buildUserSnapshot()` получает не только `user`, но и его список memberships.
6. `DlMatchResult.tgmbaseUserSnapshot` сохраняет полный снимок, включая memberships.

### Источник данных membership

Так как в `tgmbase.prisma` membership-модели пока отсутствуют, реализация должна начинаться с проверки реальной схемы `tgmbase`.

Предпочтительный путь:
- добавить в `tgmbase.prisma` недостающие модели membership-таблиц
- сгенерировать Prisma client
- использовать typed Prisma lookup

Fallback-путь, только если схема неудобна для быстрого моделирования:
- использовать `prisma.$queryRaw` для bulk lookup membership-данных

Для первой версии рекомендуется typed Prisma, если membership-таблицы читаемы и стабильны.

## Snapshot и DTO

### Snapshot пользователя

`tgmbaseUserSnapshot` расширяется полем `memberships`.

Пример структуры:

```json
{
  "user_id": "123456",
  "username": "user_one",
  "phone": "+79990000001",
  "first_name": "Иван",
  "last_name": "Иванов",
  "premium": true,
  "scam": false,
  "bot": false,
  "upd_date": "2026-03-25T00:00:00.000Z",
  "memberships": [
    {
      "type": "group",
      "group_id": "10001",
      "title": "Продажа квартир Владивосток"
    },
    {
      "type": "channel",
      "group_id": "20002",
      "title": "Новости района"
    }
  ]
}
```

### API результата

`GET /api/telegram/dl-match/runs/:id/results` продолжает возвращать тот же объект результата, но поле `user` расширяется:

- было: базовые user-поля
- станет: базовые user-поля + `memberships: []`

Если у пользователя нет найденных связей, вернуть пустой массив `memberships`, а не `null`.

## Поведение фронта

### Таблица результатов

В таблицу результатов нужно добавить колонку `Связи tgmbase`.

Содержимое ячейки:
- каждая связь отдельной строкой
- формат: `<type>: <title> (<group_id>)`

Примеры:
- `group: ЖК Центральный чат (10001)`
- `supergroup: Авто рынок (20002)`
- `channel: Новости района (30003)`

Если связей нет:
- показывать `Нет данных`

### Визуальное ограничение

Так как у одного пользователя связей может быть много, строка таблицы не должна бесконтрольно раздуваться по высоте. Рекомендуется:
- рендерить список в компактном многострочном блоке
- ограничить высоту ячейки
- разрешить внутренний скролл

Важно: это только визуальное ограничение. В данных и экспорте должны оставаться все связи без урезания.

## Экспорт XLSX

`xlsx`-экспорт должен включать все связи пользователя.

Рекомендуемый формат одной колонки:
- связи склеиваются через перевод строки
- каждая строка: `<type>: <title> (<group_id>)`

Пример содержимого ячейки:

```text
group: ЖК Центральный чат (10001)
supergroup: Авто рынок (20002)
channel: Новости района (30003)
```

Это позволит сохранить полный объём данных без усложнения схемы экспорта.

## Производительность

Нельзя делать lookup membership-данных по одному пользователю на запись результата. Это мгновенно убьёт батчевый pipeline.

Правильная стратегия:
- собирать уникальные `user_id` для всего батча
- одним bulk lookup доставать membership-данные
- группировать их по `user_id` в памяти
- переиспользовать карту при построении snapshot

Это сохраняет текущий performance-подход DL-матчинга:
- batched read
- batched lookup
- batched `createMany`

## Границы первой версии

Входят:
- подключение membership-данных из `tgmbase`
- сохранение `memberships` в `tgmbaseUserSnapshot`
- возврат `memberships` в result API
- новая колонка `Связи tgmbase` на фронте
- включение связей в `xlsx`-экспорт

Не входят:
- фильтрация результатов по membership
- отдельная карточка пользователя
- отдельная нормализованная таблица membership-результатов
- lazy expand/collapse строк таблицы

## Обработка ошибок

- если membership lookup не нашёл связей, это не ошибка; сохраняется пустой массив
- если membership lookup упал системно, весь run должен считаться `FAILED`, потому что результат будет неполным
- в логах worker-а нужно отдельно писать стадию membership lookup, чтобы было видно, где именно сломался батч

## Тестирование

### Backend

- unit: snapshot пользователя включает `memberships`
- unit: batch processing подтягивает membership-данные bulk-образом, а не per-user
- unit: `getResults()` возвращает `user.memberships`
- exporter test: `xlsx` содержит связи в ожидаемом формате

### Frontend

- render test: таблица результатов показывает колонку `Связи tgmbase`
- render test: несколько связей рендерятся в одной строке
- render test: при пустом массиве показывается `Нет данных`

## Риски

- главный риск: membership-таблицы `tgmbase` могут быть названы неочевидно или вообще отсутствовать в ожидаемом виде; тогда сначала придётся актуализировать Prisma-схему
- если на одного пользователя приходится очень много связей, snapshot станет тяжелее; но это всё ещё предпочтительнее, чем повторный lookup на каждый просмотр и экспорт
- если membership lookup сделать не bulk, а точечно, производительность матчинга резко просядет

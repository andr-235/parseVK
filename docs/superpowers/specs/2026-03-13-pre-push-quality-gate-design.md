# Дизайн: локальный pre-push quality gate

## Контекст

Сейчас в репозитории есть только быстрый `pre-commit` hook:

- `.husky/pre-commit` вызывает `npx lint-staged`

При этом CI в `.github/workflows/ci.yml` делает более строгую проверку:

- `lint`
- `type-check`
- `format-check`
- backend/frontend tests

Из-за этого локально можно закоммитить код, который выглядит приемлемо после autofix, но падает уже в CI.

## Проблема

Нужен локальный gate перед `git push`, который заранее ловит типичные CI-падения:

- форматирование;
- lint;
- type-check;
- тесты;

и делает это только для затронутых частей проекта, чтобы время ожидания оставалось разумным.

## Цель

Перед `git push` автоматически запускать строгий набор проверок, максимально близкий к CI, но только для изменённых частей репозитория.

Ожидаемое поведение:

- если изменён только `api`, проверяется только backend;
- если изменён только `front`, проверяется только frontend;
- если изменены общие конфиги или инфраструктура, проверяются обе части;
- если изменены только docs, push не блокируется тяжёлыми проверками.

## Не цели

- Не превращать `pre-push` в полный клон CI для каждого push.
- Не делать silent autofix на этапе `push`.
- Не строить сложную систему кэшей и параллельных оркестраторов в первой версии.

## Требования

### Функциональные

- Добавить строгий `pre-push` hook.
- Вычислять затронутые части проекта по diff.
- Для backend запускать:
  - `format:check`
  - `lint`
  - `typecheck`
  - `test`
- Для frontend запускать:
  - `format:check`
  - `lint`
  - `type-check`
  - `test`
- Для docs-only изменений пропускать тяжёлые проверки.

### Нефункциональные

- Локальные проверки должны быть понятны по логам.
- Команды должны по возможности совпадать с CI.
- `pre-push` не должен быть "магическим": один основной скрипт, один hook.

## Подходы

### Вариант 1. Отдельный orchestration-скрипт + тонкий hook

`pre-push` вызывает один репозиториный скрипт, который:

- определяет diff;
- решает, какие части проекта затронуты;
- запускает нужные проверки.

Плюсы:

- один источник правды;
- проще тестировать и поддерживать;
- меньше дублирования логики в hook.

Минусы:

- нужно один раз аккуратно собрать сценарии diff.

### Вариант 2. Вся логика прямо в `.husky/pre-push`

Плюсы:

- быстро внедрить.

Минусы:

- bash-логика быстро разрастается;
- хуже читается и отлаживается;
- выше риск рассинхронизации с CI.

### Вариант 3. Всегда запускать обе части целиком

Плюсы:

- минимальный риск пропустить связь между изменениями.

Минусы:

- слишком медленно;
- разработчики начнут обходить hook через `--no-verify`.

### Решение

Выбрать вариант 1: отдельный orchestration-скрипт + тонкий `pre-push`.

## Архитектура

### Общая схема

- `.husky/pre-commit` остаётся быстрым и занимается autofix только по staged файлам.
- `.husky/pre-push` вызывает один корневой скрипт, например `scripts/pre-push-check.sh`.
- Скрипт вычисляет базу сравнения для push и список изменённых файлов.
- На основе changed files определяет флаги:
  - `backend_changed`
  - `frontend_changed`
  - `docs_only`
  - `shared_config_changed`

Далее скрипт решает, какие project checks запускать.

### Логика выбора

#### Backend

Запускать backend checks, если изменён хотя бы один из путей:

- `api/**`
- `create-session.js`
- общие backend-конфиги, влияющие на backend-пайплайн

#### Frontend

Запускать frontend checks, если изменён хотя бы один из путей:

- `front/**`
- общие frontend-конфиги, влияющие на frontend-пайплайн

#### Оба проекта

Запускать оба набора, если изменены:

- root `package.json`
- `.github/workflows/ci.yml`
- `.husky/**`
- `.lintstagedrc.js`
- общие `scripts/**`
- lockfiles

#### Docs-only

Если изменены только:

- `docs/**`
- `*.md`

то строгий pre-push можно завершать успешно без запуска project checks.

## Команды проверок

### Backend

Использовать package scripts:

- `bun run format:check`
- `bun run lint`
- `bun run typecheck`
- `bun run test`

### Frontend

Frontend нужно выровнять с CI. Лучше завести явный script:

- `typecheck: "tsc --noEmit"`

И затем использовать:

- `bun run format:check`
- `bun run lint`
- `bun run typecheck`
- `bun run test`

Это убирает дублирование между hook и CI.

## Политика autofix

### `pre-commit`

Оставить autofix:

- форматирование по staged файлам;
- eslint `--fix` по изменённым файлам там, где это уже используется.

### `pre-push`

Не делать silent autofix.

Причина:

- на этапе push неожиданные изменения файлов неудобны;
- если `push` меняет рабочее дерево, появляется риск запушить код, который разработчик не посмотрел;
- строгий gate должен fail-fast, а не переписывать код.

Если локальная проверка не проходит, разработчик должен явно запустить исправление и закоммитить результат.

## Выравнивание с CI

Чтобы локальная и удалённая проверки меньше расходились, нужно по возможности опереться на одинаковые package scripts.

Предлагаемая структура:

- `api/package.json`:
  - `typecheck`
  - `check:push`
- `front/package.json`:
  - `typecheck`
  - `check:push`

Где `check:push` последовательно вызывает:

- `format:check`
- `lint`
- `typecheck`
- `test`

Тогда:

- `pre-push` вызывает `bun run check:push` для затронутых проектов;
- CI тоже можно постепенно перевести на те же scripts.

## Минимальный scope первой версии

- Один `pre-push` hook.
- Один orchestration-скрипт в корне проекта.
- Явный `front:typecheck` script.
- Явные `check:push` scripts для `api` и `front`.
- Логика diff без параллельного запуска и без сложного кэширования.

## Риски

- Если логика определения changed files будет слишком грубой, pre-push может гонять лишнее.
- Если логика будет слишком узкой, можно пропустить важные проверки.
- Слишком медленный pre-push может спровоцировать обход hook.

Поэтому первая версия должна быть:

- строгой, но не чрезмерно умной;
- с понятной таблицей правил по путям;
- с чистым fail-fast поведением.

## Критерии успеха

- Перед `git push` локально ловятся типичные format/lint/type/test ошибки.
- Поведение pre-push предсказуемо и прозрачно по логам.
- CI начинает падать заметно реже по базовым quality checks.
- Разработчику не нужно вручную вспоминать, какие команды запускать перед push.

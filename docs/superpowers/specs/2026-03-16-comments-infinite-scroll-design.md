# Дизайн: стабилизация infinite scroll и счётчиков на странице комментариев

## Контекст

На странице комментариев в обычном режиме без поисковой строки список визуально ведёт себя как один непрерывный поток: комментарии быстро догружаются автоматически, а числа в разных частях UI воспринимаются как противоречивые. При этом текущая архитектура уже использует `cursor`-пагинацию и infinite scroll, поэтому задача не в смене паттерна загрузки, а в стабилизации его поведения и выравнивании счётчиков.

Симптомы укладываются в две связанные проблемы:

- auto-load может срабатывать несколько раз подряд сразу после первого рендера, если sentinel уже находится во viewport;
- UI смешивает серверные счётчики с локально вычисленными числами после фильтрации, группировки и категоризации.

## Цели

- Сохранить текущий infinite scroll на `cursor`-пагинации.
- Убрать ощущение, что список грузится "одним потоком" без контролируемых порций.
- Разделить серверные и клиентские счётчики, чтобы пользователь видел согласованные числа.
- Сохранить совместимость с текущими фильтрами по статусу, ключевым словам и категориям.

## Не цели

- Не переводить страницу на классическую пагинацию с номерами страниц.
- Не делать широкий рефакторинг всей comments-архитектуры без необходимости.
- Не менять search flow, если проблема воспроизводится в обычном режиме без поиска.

## Подходы

### Вариант 1. Сохранить cursor + infinite scroll и развести источники правды

Оставить текущую схему загрузки, но сделать явный контракт данных: API/store отвечают за глобальные счётчики и пагинацию, UI считает только локальные производные метрики. Дополнительно ограничить цепочку auto-load после initial render.

Плюсы:

- Минимальное вмешательство в архитектуру.
- Закрывает оба симптома: автодогрузку и путаницу в числах.
- Не ломает существующий UX со скроллом.

Минусы:

- Требует аккуратно пройти по всем местам, где сейчас смешаны значения `visible`, `loaded` и `total`.

### Вариант 2. Починить только подписи и отображение чисел

Оставить текущую механику загрузки и изменить только тексты, бейджи и подписи в карточках.

Плюсы:

- Самый дешёвый по объёму.

Минусы:

- Не решает корень проблемы, если auto-load действительно запускает несколько запросов подряд.
- Оставляет риск дрожащих или плавающих счётчиков.

### Вариант 3. Полностью переорганизовать orchestration загрузки comments

Свести `useCommentsQuery`, store и load-more flow к более строгой state machine с единым оркестратором.

Плюсы:

- Самая чистая архитектурно схема.

Минусы:

- Непропорционально большой объём для текущего дефекта.
- Повышенный риск регрессий в уже рабочем comments flow.

## Выбранное решение

Использовать вариант `1`.

Сохраняем `cursor`-пагинацию и infinite scroll, но явно разделяем:

- серверные счётчики и состояние пагинации;
- клиентские производные счётчики для конкретного представления списка.

Параллельно ограничиваем auto-load так, чтобы после initial render допускалась только контролируемая догрузка, а не каскад запросов без участия пользователя.

## Архитектура данных

### Источник правды

`API /comments/cursor` и `commentsStore` остаются единственным источником правды для:

- `totalCount`
- `readCount`
- `unreadCount`
- `hasMore`
- `nextCursor`

### Производные UI-метрики

UI считает отдельно и не смешивает с серверными значениями:

- `loadedCount` — сколько комментариев реально накоплено в store;
- `renderedCount` — сколько комментариев реально отображается после локальной группировки и category filtering;
- `keywordCommentsTotal` — локальный счётчик комментариев с keyword match.

### Правило отображения

Если `loadedCount < totalCount`, интерфейс должен явно показывать, что это частично загруженный набор, а не полный список. Формулировки вида `visibleCount / totalCount`, где левое число уже прошло дополнительные клиентские преобразования, считаются неоднозначными и подлежат замене на явные подписи.

## Поведение infinite scroll

### Initial load

- Первая страница загружается через `cursor`.
- Если sentinel сразу во viewport, допускается максимум один автоматический `load more`, чтобы не оставлять экран полупустым.

### Дальнейшая догрузка

- После этого новые порции подгружаются только при новом пересечении sentinel или явном пользовательском скролле вниз.
- Каскад нескольких auto-load подряд без нового пользовательского действия не допускается.

### Фоновая синхронизация

`useCommentsQuery` может обновлять первую страницу и общие счётчики, но не должен:

- терять уже накопленные страницы;
- произвольно сбрасывать `nextCursor`;
- переводить `hasMore` в неконсистентное состояние;
- визуально создавать эффект полной перезагрузки всей ленты.

## Компонентные границы

### `commentsStore`

Отвечает за:

- накопленный список comments;
- `nextCursor`, `hasMore`;
- серверные счётчики;
- reset при смене серверных фильтров.

### `useCommentsQuery`

Отвечает за:

- initial sync;
- background refresh первой страницы и счётчиков;
- аккуратное слияние данных без потери уже загруженных элементов.

### `CommentsTableCard`

Отвечает за:

- рендер карточек и группировок;
- отображение явно подписанных счётчиков;
- вызов `onLoadMore` только в пределах ожидаемого auto-load поведения.

## Обработка ошибок

- Ошибка initial load показывает стандартное сообщение загрузки comments.
- Ошибка `load more` не должна сбрасывать уже загруженный список.
- Ошибка фонового sync не должна ломать текущую ленту, если данные уже есть локально.

## Тестирование и верификация

- Без поиска, только с фильтрами, initial load не уходит в бесконечную цепочку запросов.
- `loadedCount` растёт по страницам, `totalCount` не подменяется локальными числами.
- После смены фильтров store корректно reset-ится и запускает новую cursor-цепочку.
- При переключении `read`, `unread`, `all` серверные счётчики согласованы со списком.
- Background refetch не теряет уже загруженные страницы.
- UI явно различает "всего", "загружено" и "показано".

## Риски

- Главный риск: не все текущие места использования счётчиков будут переведены на новый контракт, и путаница частично останется.
- Второй риск: слишком агрессивное ограничение auto-load может ухудшить UX на больших экранах, если первая страница окажется слишком короткой.
- Третий риск: merge-логика `useCommentsQuery` и store может скрыто дублировать или затирать состояние при race condition.

## Итог

Страница комментариев сохраняет текущий infinite scroll, но получает более строгий контракт: сервер отвечает за глобальные счётчики и пагинацию, клиент отдельно считает показатели отображения. Infinite scroll остаётся автоматическим, но перестаёт бесконтрольно каскадировать запросы сразу после рендера. Это адресует текущие симптомы без смены пользовательского паттерна навигации.

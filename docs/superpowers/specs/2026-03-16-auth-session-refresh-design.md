# Дизайн: устойчивое автоматическое продление авторизации

Дата: 2026-03-16

## Контекст

В проекте уже реализована JWT-аутентификация с `accessToken` и `refreshToken`. Frontend хранит оба токена в `zustand persist`, использует `accessToken` для обычных API-запросов и пытается обновить его через `POST /auth/refresh` при получении `401`.

По фактическому поведению пользователь может быть резко выброшен на страницу логина даже во время активной работы в интерфейсе. После перезагрузки страницы сессия не восстанавливается, что указывает не только на истечение `accessToken`, но и на потерю клиентом действующей refresh-сессии.

## Проблема

Текущий сценарий авторизации недостаточно устойчив к рабочим ошибкам:

- продление access token происходит реактивно, только после `401`;
- неуспешный refresh сразу очищает весь auth store;
- временные сетевые или серверные ошибки не отделяются от реально невалидного refresh token;
- при ротации refresh token возможны гонки между параллельными запросами.

В результате пользователь теряет сессию раньше, чем это необходимо по бизнес-логике.

## Цели

- Сохранять авторизацию во время обычной активной работы пользователя.
- Продлевать access token автоматически до его истечения.
- Разлогинивать пользователя только при реальной потере refresh-сессии.
- Избежать ложного logout из-за кратковременной сетевой ошибки или единичного `5xx`.
- Сохранить текущую архитектуру JWT без перехода на отдельную cookie-сессию.

## Не входит в текущий объём

- Полный переход на cookie-only auth flow.
- Изменение бизнес-правил TTL токенов.
- Перестройка роутинга приложения.
- Введение отдельного серверного session store сверх текущей схемы refresh token hash.

## Подход

Рекомендованный вариант: усилить существующую схему JWT refresh, не меняя базовую архитектуру.

Ключевые принципы:

- access token обновляется превентивно, до истечения;
- refresh-on-401 остаётся резервным механизмом;
- logout выполняется только при явном подтверждении невалидного refresh token;
- временные ошибки refresh не очищают сессию сразу;
- параллельные запросы используют единый in-flight refresh, чтобы не ломать ротацию токенов.

## Архитектура решения

### Frontend

Основная точка управления сессией остаётся в auth-модуле frontend.

Планируемые изменения:

- `AuthProvider` получает планировщик следующего refresh по `exp` access token;
- после логина и после каждого успешного refresh планировщик запускается заново;
- refresh выполняется заранее, с запасом до реального истечения токена;
- `createRequest()` продолжает поддерживать refresh после `401`, если превентивное продление не успело сработать;
- `refreshAccessToken()` различает тип ошибки и очищает store только при подтверждённой невалидности refresh token.

### Backend

Backend сохраняет текущий endpoint `POST /auth/refresh` и ротацию refresh token.

Изменения на backend ограничены проверкой устойчивости текущей реализации:

- корректная работа последовательных refresh-вызовов;
- корректное поведение при конкурентных запросах вокруг истечения access token;
- сохранение единообразного ответа `401` для реально невалидного refresh token.

Если потребуется, допустима небольшая правка тестов или вспомогательной логики, но без изменения общего контракта auth API.

## Поток данных

### Нормальный рабочий цикл

1. Пользователь логинится и получает `accessToken` и `refreshToken`.
2. Frontend сохраняет токены в auth store.
3. `AuthProvider` декодирует `exp` access token и планирует refresh заранее.
4. До истечения access token frontend вызывает `/auth/refresh`.
5. Backend возвращает новую пару токенов.
6. Frontend атомарно обновляет store и перепланирует следующий refresh.

### Резервный сценарий через `401`

Если обычный API-запрос вернул `401`:

1. `createRequest()` запускает `refreshAccessToken()`;
2. при успехе исходный запрос автоматически повторяется с новым access token;
3. пользователь остаётся в текущем экране без перехода на логин.

### Сценарий неуспешного refresh

Поведение делится на два класса:

- явный `401` или `403` от `/auth/refresh`: refresh token считается недействительным, auth store очищается, переход на логин допустим;
- сетевой сбой, timeout или `5xx`: сессия временно сохраняется, logout не выполняется немедленно.

Такой подход не путает инфраструктурную проблему с реальной потерей авторизации.

## Управление гонками

Ротация refresh token делает систему чувствительной к параллельным refresh-запросам. Поэтому frontend должен использовать единый shared `refreshPromise` как точку синхронизации.

Требования:

- если refresh уже выполняется, новые запросы не запускают второй refresh;
- все ожидающие запросы получают результат из текущего in-flight refresh;
- после успешного refresh store обновляется один раз последней валидной парой токенов;
- повторное планирование таймера происходит только после завершения текущего refresh.

## Обработка ошибок

### Ошибки, которые не должны ломать сессию сразу

- временное отсутствие сети;
- кратковременный `502/503/504`;
- непредвиденный fetch error на клиенте;
- кратковременная недоступность backend.

В этих случаях система:

- не вызывает `clearAuth()` немедленно;
- оставляет пользователя в приложении;
- позволяет повторить refresh позже по следующему запросу или следующему таймеру.

### Ошибки, которые должны завершать сессию

- `401 Unauthorized` от `/auth/refresh`;
- `403 Forbidden` от `/auth/refresh`, если backend использует такой код для отклонённого refresh;
- отсутствие refresh token в store;
- повреждённый или заведомо невалидный refresh token.

В этих случаях очистка auth store является корректной.

## Тестирование

Нужны проверки нескольких уровней.

### Frontend unit

- вычисление времени следующего refresh по `exp`;
- поведение `refreshAccessToken()` при `401`;
- поведение `refreshAccessToken()` при `5xx` и сетевой ошибке;
- отсутствие повторного параллельного refresh при уже активном `refreshPromise`.

### Frontend integration

- access token истекает во время активной работы, refresh успешен, пользователь остаётся в приложении;
- обычный запрос получает `401`, затем успешно повторяется после refresh;
- временный сбой refresh не выбрасывает пользователя на логин мгновенно.

### Backend tests

- refresh с валидным токеном возвращает новую пару токенов;
- повторное использование устаревшего refresh token отклоняется;
- последовательные refresh-операции работают корректно;
- обработка конкурентных refresh-вызовов соответствует ожидаемой политике ротации.

## Риски

- Если оставить logout на любой ошибке refresh, пользователь продолжит терять сессию даже после частичных правок.
- Если превентивный refresh будет запускаться слишком поздно, часть запросов всё равно будет падать в `401`.
- Если запустить несколько refresh одновременно, ротация refresh token может инвалидировать предыдущий токен и вызвать ложный logout.
- Если backend не различает типы отказов достаточно явно, фронту придётся опираться на HTTP-статус и контракт ответа.

## Критерии готовности

- Пользователь не выбрасывается на логин во время обычной активной работы при валидном refresh token.
- Access token продлевается автоматически до истечения.
- `401` на обычном API не приводит к logout, если refresh проходит успешно.
- Временная ошибка refresh не очищает auth store мгновенно.
- Logout происходит только при реальной невалидности refresh token.
- После успешного refresh приложение использует новую пару токенов и продолжает работу без ручного логина.

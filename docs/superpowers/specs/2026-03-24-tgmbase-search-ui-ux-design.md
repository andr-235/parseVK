# Tgmbase Search UI/UX Design

**Контекст**

Текущая страница `tgmbase-search` решает задачу пакетного поиска, но UX плохо масштабируется на реальные батчи `100+` запросов. После запуска пользователь получает длинный линейный экран: форма, прогресс, summary-таблица и затем все карточки результатов подряд. Это создаёт лишний скролл, мешает быстро разбирать проблемные записи и затрудняет переход между обзором батча и детальным просмотром конкретного результата.

Ключевая потребность пользователя двойная:
- быстро фильтровать и разбирать массовый батч после завершения поиска
- детально просматривать конкретную запись без потери общего контекста

**Цель**

Перестроить `tgmbase-search` в рабочий экран для больших батчей: компактный запуск поиска, липкий обзор прогресса и summary, мощные фильтры и список результатов в формате `master-detail`, где пользователь быстро перемещается между строками, а детали выбранной записи остаются рядом.

## Рекомендованный подход

Рекомендуется подход `master-detail`.

Альтернативы рассматривались:
- улучшение текущей линейной страницы с фильтрами над длинным списком карточек
- разделение на два экрана: обзор батча и отдельный детальный просмотр

Обе альтернативы уступают по скорости ручного разбора батчей `100+`. Линейный экран сохраняет проблему длинного скролла, а двухэкранный сценарий создаёт лишние переходы. `Master-detail` лучше всего объединяет обзор батча и быстрый просмотр деталей.

## Структура экрана

Страница делится на четыре зоны.

### 1. Sticky-блок запуска

Верхний блок содержит:
- `textarea` для пакетного ввода
- счётчик строк
- краткие правила формата: `telegramId`, `@username`, телефон
- действия `Очистить`, `Найти`, `Новый батч`

До запуска это основной блок ввода. После запуска и после получения результата блок не исчезает, а сворачивается в более компактный sticky-вид, чтобы не занимать лишнюю высоту и при этом сохранять быстрый доступ к повторному запуску.

### 2. Sticky-панель обзора батча

Под формой располагается липкий блок со сводкой:
- всего
- найдено
- не найдено
- несколько совпадений
- невалидные
- ошибки

Каждый summary-показатель кликабелен и работает как быстрый фильтр. Например, нажатие на `Не найдено` сразу ограничивает список только этими строками.

### 3. Основная рабочая область `master-detail`

На десктопе страница делится на две колонки:
- слева список результатов батча
- справа детальная панель выбранной записи

Пользователь больше не листает сотни карточек вниз. Он работает со списком, а подробности текущей строки видит рядом.

### 4. Адаптивный режим

Поведение по брейкпоинтам:
- desktop: две колонки `master-detail`
- tablet: список сверху, детали снизу
- mobile: детали открываются как отдельная вкладка или нижняя панель поверх списка

Это сохраняет единый сценарий работы без перегрузки маленьких экранов.

## Пользовательский поток

1. Пользователь вставляет `100+` запросов.
2. Запускает поиск.
3. Во время выполнения видит live-прогресс и статус соединения.
4. После завершения сразу получает summary и отфильтрованный рабочий список.
5. Включает нужные фильтры.
6. Выбирает строку.
7. Изучает детали справа и быстро переключается на следующую запись кликом или клавиатурой.

## Фильтры и управление

Панель управления должна быть двухуровневой.

### Быстрые фильтры

Всегда видимая строка статусных чипов:
- `Все`
- `Найдено`
- `Не найдено`
- `Несколько`
- `Ошибка`
- `Невалидный`

Каждый чип содержит счётчик элементов в текущем батче.

### Дополнительные фильтры

Вторая строка или выпадающий блок `Ещё фильтры` содержит:
- тип запроса: `telegramId`, `username`, `phone`
- наличие данных: `есть профиль`, `есть чаты`, `есть контакты`, `есть сообщения`

### Текстовый поиск

Один глобальный input фильтрует список по:
- `query`
- `normalizedQuery`
- `fullName`
- `username`
- `phone`
- `telegramId`
- названиям чатов

### Сортировка

Поддерживаются режимы:
- по умолчанию: проблемные записи выше успешных
- по порядку ввода
- по статусу
- по числу сообщений
- по числу контактов
- по числу чатов

### Служебные действия

Нужны действия:
- `Сбросить фильтры`
- `Показать только выбранное`
- `Следующий результат`
- `Предыдущий результат`
- копирование идентификаторов из строки или панели деталей

## Список результатов

Левая колонка должна быть ориентирована на быстрое визуальное сканирование, а не на полную детализацию.

Каждая строка показывает:
- исходный `query`
- тип запроса
- локализованный статус с цветовой маркировкой
- главное имя или первый кандидат
- короткие счётчики: чаты, контакты, сообщения
- индикатор наличия профиля
- индикатор множественных кандидатов

Поведение:
- один клик выбирает строку
- текущая строка заметно выделена
- клавиши вверх/вниз переключают активную запись
- при больших батчах список должен поддерживать виртуализацию

## Панель деталей

Правая колонка показывает только выбранную запись и делится на смысловые секции.

### Обзор

Содержит:
- статус
- исходный запрос
- нормализованное значение
- профиль пользователя
- основные идентификаторы

Если по записи найдено несколько кандидатов, в верхней части панели появляется компактный блок выбора кандидата.

### Чаты

Показывает найденные чаты и каналы:
- `title`
- `type`
- `peerId`

### Контакты

Показывает активные контакты:
- `fullName`
- `username` или `telegramId`
- число общих чатов
- число сообщений

### Сообщения

Показывает ленту сообщений в отдельной секции с `load more`. Сообщения больше не должны быть частью длинной общей карточки всего списка.

## Состояния интерфейса

### До запуска

Страница показывает:
- пример формата ввода
- число строк
- предвалидацию подозрительных строк

Если часть строк явно не похожа на поддерживаемый формат, пользователь видит предупреждение ещё до отправки.

### Во время поиска

Прогресс-блок показывает:
- процент
- `обработано X из Y`
- батч `N из M`
- статус websocket
- активное состояние выполнения

Экран не должен выглядеть зависшим даже при долгой обработке больших батчей.

### После завершения

Summary остаётся видимым. Список автоматически сортируется так, чтобы проблемные записи были сверху. Автовыбор первой строки работает по правилу:
- сначала первая проблемная запись
- если проблемных нет, первая найденная запись

### Частичные ошибки

Ошибки не теряются в общей строке текста. Они участвуют в summary и фильтрах наравне с остальными статусами.

### Пустой результат

Если совпадений нет, экран всё равно показывает summary и пустой список с объяснением, а не только один большой `EmptyState`.

### Новый запуск

Новый запуск полностью заменяет предыдущий батч на экране. История запусков в первую версию не входит.

## Технический дизайн

Изменение должно строиться поверх существующего `useTgmbaseSearchState`, без полного переписывания модуля.

- `front/src/modules/tgmbase-search/components/TgmbaseResultDetails.tsx`

- либо адаптированы под новые роли
- либо частично заменены более узкими компонентами

## UI State

Помимо базового результата поиска, нужен отдельный слой локального UI-state:
- `selectedQuery`
- `searchTerm`
- `statusFilter[]`
- `queryTypeFilter[]`
- `hasProfile`
- `hasGroups`
- `hasContacts`
- `hasMessages`
- `sortBy`
- `detailsTab`

Фильтрация и сортировка считаются как derived state поверх `response.items`.

Правила выбора:
- если выбранная запись исчезает из-за фильтра, выбирается первая видимая
- если видимых записей нет, правая панель показывает пустое состояние с действием `Сбросить фильтры`

## Ограничение первой версии

В первую итерацию не включаются:
- история батчей
- экспорт отфильтрованных результатов
- пользовательские пресеты фильтров

Это полезные функции, но они не нужны для резкого улучшения UX текущего экрана.

## Тестирование

Нужны фронтенд-тесты на:
- фильтрацию по статусу
- комбинированные фильтры
- текстовый поиск
- сортировку
- автовыбор первой видимой записи
- корректное пустое состояние при отсутствии видимых строк
- сохранение выбора при смене сортировки
- отсутствие рендера тяжёлой детализации для всех строк одновременно

Отдельно стоит проверить адаптивное поведение ключевых блоков и базовую клавиатурную навигацию по списку.

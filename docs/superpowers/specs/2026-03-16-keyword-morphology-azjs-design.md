# Дизайн: морфология ключевых слов через Az.js

Дата: 2026-03-16

## Контекст

Сейчас страница комментариев и pipeline `CommentKeywordMatch` опираются на точные ключевые слова из БД. Это заставляет вручную добавлять в keywords падежи и словоформы, иначе комментарии с морфологическими вариантами не попадают в matches и не доходят до страницы в ожидаемом виде.

Нужен production-контур, в котором:

- одно ключевое слово автоматически разворачивается в словоформы;
- словоформы сохраняются в БД, а не вычисляются каждый раз на лету;
- backend matching комментариев работает по полному набору форм;
- ручные формы тоже поддерживаются;
- ручные удаления autogenerated-форм не возвращаются после регенерации;
- frontend не становится вторым морфологическим движком.

В качестве морфологического движка используется `Az.js`.

## Цели

- Убрать необходимость вручную заносить в keywords все падежи и словоформы.
- Сохранить прозрачную и отлаживаемую модель данных.
- Сделать морфологию частью production pipeline, а не временным эвристическим слоем.
- Обеспечить стабильный backend matching для `CommentKeywordMatch`.
- Поддержать ручное добавление и исключение словоформ без потери автогенерации.

## Не входит в объём

- Полнотекстовый поиск и Elasticsearch.
- Синонимы и семантически близкие слова.
- Мультиязычная морфология.
- Вычисление морфологии на frontend для принятия продуктовых решений.

## Подход

Рекомендованный вариант: расширить доменную модель keywords через отдельную таблицу словоформ и общий backend-сервис морфологии на базе `Az.js`.

Архитектурный принцип:

- `Keyword` остаётся основной сущностью.
- `KeywordForm` хранит конкретные формы.
- морфология считается на backend;
- комментарии матчятся по сохранённым формам;
- ручные формы и ручные исключения живут в БД рядом с автогенерируемыми;
- frontend только отображает и редактирует формы, но не решает сам, что является совпадением.

## Модель данных

### Основные сущности

#### Keyword

Сохраняет исходное ключевое слово и текущую бизнес-метаинформацию.

#### KeywordForm

Одна запись = одна словоформа keyword.

Предлагаемые поля:

- `id`
- `keywordId`
- `form`
- `source` = `generated | manual`
- `createdAt`
- `updatedAt`

#### KeywordFormExclusion

Хранит исключения для autogenerated-форм, которые пользователь явно отключил.

Предлагаемые поля:

- `id`
- `keywordId`
- `form`
- `createdAt`

### Почему не JSON

JSON в `Keyword` проще внедрить, но хуже подходит для production-варианта, потому что:

- его сложнее индексировать;
- сложнее контролировать дифф между generated/manual формами;
- неудобнее строить понятные Prisma-запросы;
- сложнее поддерживать исключения и отладку.

Отдельные таблицы лучше соответствуют целям проекта: прозрачность, расширяемость и контроль.

## Генерация словоформ

### Источник

Словоформы генерируются через `Az.js`.

### Когда генерировать

- при создании keyword;
- при обновлении основного слова keyword;
- при явной операции `перегенерировать формы`;
- при bulk-upload keywords.

### Правила регенерации

- пересчитываются только `generated` формы;
- `manual` формы не удаляются;
- `KeywordFormExclusion` применяется после генерации и не даёт вернуть вручную исключённые autogenerated-формы;
- итоговый набор форм должен быть дедуплицирован и нормализован.

## Matching pipeline

### Источник кандидатов

При подготовке keyword candidates backend загружает:

- основное слово `Keyword.word`;
- все `KeywordForm`;
- исключённые generated-формы не учитываются, потому что они уже отфильтрованы в модели.

### Логика матчинга

Для каждого keyword строится полный набор форм:

- исходное слово;
- generated-формы;
- manual-формы.

Каждая форма проходит ту же нормализацию, которая используется в текущем pipeline keyword-matching.

Далее:

- текст комментария нормализуется;
- поиск совпадений выполняется уже не по одному слову, а по набору словоформ keyword;
- при нахождении совпадения создаётся `CommentKeywordMatch`, как и сейчас.

### Поведение совпадения

Для текущего продукта согласовано более широкое правило:

- совпадение допускается не только по отдельному токену;
- допустимы частичные вхождения, если это соответствует текущему поведению keyword matching;
- морфология расширяет набор допустимых форм, а не заменяет существующий принцип сравнения.

## Использование на странице комментариев

Страница комментариев не должна сама вычислять морфологию.

Production-истина должна оставаться в backend:

- backend создаёт `CommentKeywordMatch`;
- frontend получает уже готовые matches;
- отображение категорий, keywords и фильтров строится из сохранённых данных;
- ручной поисковый ввод на странице может быть отдельно улучшен позже, но не должен становиться вторым источником морфологического решения.

## Админка keywords

Для keywords нужен production-контур управления формами.

### Что должен видеть оператор

- основное слово;
- generated-формы;
- manual-формы;
- исключённые autogenerated-формы.

### Что должен уметь оператор

- добавить manual-форму;
- удалить manual-форму;
- исключить autogenerated-форму;
- перегенерировать формы для keyword;
- запустить пересчёт matches после изменения форм.

### Инварианты

- `generated` формы управляются системой;
- `manual` формы управляются пользователем;
- удалённая autogenerated-форма не должна возвращаться после регенерации;
- изменение keyword forms должно приводить к предсказуемому пересчёту `CommentKeywordMatch`.

## Операции пересчёта

После изменения keyword forms нужны надёжные серверные операции:

- пересчёт matches для одного keyword;
- пересчёт matches для набора keywords после bulk-upload;
- пересчёт всех matches, если изменится логика морфологии или нормализации.

Эти операции должны быть явными, наблюдаемыми и безопасными для production.

## Backend-архитектура

Нужны отдельные компоненты:

- `KeywordMorphologyService` для генерации форм через `Az.js`;
- сервис синхронизации `KeywordForm`;
- обновлённый loader keyword candidates для comments pipeline;
- обновлённый matcher комментариев;
- сервис пересчёта keyword matches после изменения forms.

Важно не встраивать всю логику прямо в один большой `CommentsSaverService`, иначе морфология быстро станет трудно поддерживаемой.

## Риски

- `Az.js` может давать неидеальные формы для некоторых слов и фраз, это надо валидировать на реальных keywords.
- Частичные вхождения повышают риск ложных совпадений.
- Массовый пересчёт `CommentKeywordMatch` может быть дорогим на больших объёмах.
- Без хранения исключений autogenerated-форм ручные правки будут теряться.

## Почему не Elasticsearch

Elasticsearch решает другую задачу: поиск и ранжирование.

В этой задаче основной смысл не в поиске, а в расширении набора словоформ для существующего keyword-matching pipeline. Поэтому правильный production-слой находится в домене keywords/comments, а не в отдельном search-engine.

## Критерии успеха

- Оператор добавляет одно базовое ключевое слово без ручного перечисления падежей.
- Комментарии с морфологическими вариантами этого слова получают `CommentKeywordMatch`.
- Формы хранятся прозрачно и доступны для ручной правки.
- Ручные формы и исключения не теряются после регенерации.
- Страница комментариев показывает уже корректные matches из backend.

# Telegram DL Match Chat Activity Design

**Контекст**

Сейчас DL-матчинг находит пользователей `tgmbase` по трём сигналам:
- `telegramId`
- `username`
- `phone`

После этого результат дополнительно показывает связанные чаты `tgmbase`, построенные по активности пользователя в `message.from_id -> message.peer_id`. Эти чаты и сообщения пока носят справочный характер и не влияют на сам факт попадания строки в выдачу.

Пользователь хочет изменить модель:
- чаты активности и комментарии должны стать частью логики результата
- нужно уметь исключать конкретный `peer_id`
- после исключения должны исчезать строки, которые держались только на этом чате
- в интерфейсе нужно показывать все комментарии пользователя по найденным чатам

**Цель**

Сделать `chatActivityMatch` полноценным типом совпадения, сохранить все найденные чаты и все сообщения пользователя по ним, а также добавить механизм исключения `peer_id`, который пересчитывает активные результаты последнего запуска без полного повторного матчинга по всей DL-базе.

## Рекомендованный подход

Рекомендуется разделить результат матчинга на три слоя:
- основная строка результата `dl_match_result`
- связанные чаты результата `dl_match_result_chat`
- сообщения результата `dl_match_result_message`

Исключения нужно хранить на уровне связанных чатов результата, а не пересчитывать весь run с нуля.

Рассмотренные альтернативы:
- полный повторный матчинг после каждого исключения
- чисто frontend-фильтр без изменения backend-данных
- хранение всех сообщений внутри JSON snapshot

Полный повторный матчинг слишком дорогой для объёма DL. Frontend-фильтр ломает правдивость summary и экспорта. Хранение всех сообщений в JSON snapshot быстро раздует `dl_match_result` и усложнит выборки.

## Новая модель совпадений

### Типы сигналов

Результат может быть активен по четырём типам:
- `strictTelegramIdMatch`
- `usernameMatch`
- `phoneMatch`
- `chatActivityMatch`

### Что такое `chatActivityMatch`

`chatActivityMatch` означает, что найденный пользователь `tgmbase` имеет хотя бы один активный чат результата, в котором он писал сообщения, и этот чат не исключён.

Источник данных:
- `message.from_id = tgmbase user.user_id`
- `message.peer_id` резолвится в `group`, `supergroup` или `channel`

Это не membership-реестр, а activity-based signal по сообщениям.

### Правило видимости результата

Строка остаётся в выдаче, если после учёта исключений у неё активен хотя бы один сигнал:
- `ID`
- `username`
- `phone`
- `chatActivity`

Если после исключения чатов не осталось ни одного активного сигнала, строка полностью исчезает из выдачи.

## Новые таблицы результата

### `dl_match_result_chat`

Одна запись = один найденный чат в рамках конкретного `dl_match_result`.

Поля:
- `id`
- `result_id`
- `peer_id`
- `chat_type` (`group`, `supergroup`, `channel`)
- `title`
- `is_excluded`
- `created_at`
- `updated_at`

Назначение:
- хранить все чаты, на которых держится `chatActivityMatch`
- быстро исключать/возвращать `peer_id`
- строить UI и экспорт без повторного lookup в `tgmbase`

### `dl_match_result_message`

Одна запись = одно сообщение найденного пользователя в одном найденном чате результата.

Поля:
- `id`
- `result_id`
- `peer_id`
- `message_id`
- `message_date`
- `text`
- `created_at`

Назначение:
- хранить все комментарии пользователя по найденным чатам
- убирать сообщения исключённого `peer_id` из активной выдачи без потери исходных данных

## Изменения в `dl_match_result`

`dl_match_result` сохраняет:
- текущие сигналы `ID`, `username`, `phone`
- новый флаг `chatActivityMatch`
- snapshot DL-контакта
- snapshot пользователя `tgmbase`

`relatedChats` и полные тексты комментариев больше не являются главным источником данных в snapshot. Snapshot можно оставить с лёгким summary, но канонические данные результата должны жить в новых таблицах.

## Архитектура backend

### Построение run

Во время фонового матчинга worker делает следующее:

1. Находит совпадения по `telegramId`, `username`, `phone`.
2. Для всех найденных пользователей батча bulk-образом загружает их активность из `message`.
3. Резолвит `peer_id` в `group` / `supergroup` / `channel`.
4. Создаёт `dl_match_result`.
5. Создаёт связанные записи в `dl_match_result_chat`.
6. Создаёт связанные записи в `dl_match_result_message`.
7. Выставляет `chatActivityMatch = true`, если у результата есть хотя бы один найденный чат активности.

### Пересчёт после исключения

Полный DL-match повторно не запускается.

После исключения `peer_id` backend:

1. Находит все `dl_match_result_chat` для run и этого `peer_id`.
2. Помечает их `is_excluded = true`.
3. Пересчитывает `chatActivityMatch` у затронутых `dl_match_result`.
4. Считает, остался ли у результата хотя бы один активный сигнал.
5. Если сигналов не осталось, результат перестаёт попадать в `GET /results` и `export`.

Аналогично `DELETE` для исключения возвращает чат и запускает обратный быстрый пересчёт.

## API

### `GET /api/telegram/dl-match/runs/:id/results`

Возвращает только активные результаты с учётом исключённых чатов.

Для каждой строки возвращаются:
- базовые поля результата
- флаги `ID/username/phone/chatActivity`
- активные чаты результата
- агрегированная информация по сообщениям или признак наличия сообщений

### `GET /api/telegram/dl-match/runs/:id/results/:resultId/messages`

Возвращает все сообщения результата с группировкой по `peer_id` и признаком `isExcluded`.

Минимальные поля:
- `peerId`
- `chatType`
- `title`
- `isExcluded`
- `messages[]`

У сообщения:
- `messageId`
- `messageDate`
- `text`

### `POST /api/telegram/dl-match/runs/:id/excluded-chats`

Тело:

```json
{
  "peerId": "123456789"
}
```

Действие:
- исключает чат
- пересчитывает затронутые результаты
- возвращает обновлённую сводку запуска

### `DELETE /api/telegram/dl-match/runs/:id/excluded-chats/:peerId`

Возвращает чат в активный набор и пересчитывает результаты.

### `GET /api/telegram/dl-match/runs/:id/export`

Экспортирует только активные результаты:
- без исключённых чатов
- без сообщений из исключённых чатов

## Поведение фронта

### Таблица результатов

В таблице результатов нужно добавить:
- новый badge `Chat activity match`
- колонку `Связи / действия`
- возможность исключать чат по кнопке `Исключить`

Каждая связь отображается как:
- `<type>: <title> (<peer_id>)`

Рядом:
- кнопка `Исключить`

### Комментарии

Комментарии лучше показывать не прямо в основной плотной строке таблицы, а в раскрываемом detail-view строки.

В detail-view:
- блок по каждому чату
- заголовок чата
- список всех сообщений пользователя в этом чате

Это сохраняет table-first UX и не превращает таблицу в вертикальную портянку.

### После исключения

Пользователь исключает `peer_id`, затем:
- таблица обновляется
- badge `Chat activity match` может исчезнуть
- строки без активных сигналов исчезают из выдачи
- summary запуска обновляется по активным результатам

## Экспорт XLSX

Экспорт должен учитывать исключения.

Рекомендуемый формат:
- основная строка результата
- отдельная ячейка с активными чатами
- отдельная ячейка или набор строк с комментариями пользователя по активным чатам

Ключевое правило:
- исключённые `peer_id` и их сообщения в экспорт не попадают

## Производительность

### Что нельзя делать

Нельзя:
- заново матчить всю DL-базу после каждого исключения
- загружать сообщения per-result или per-chat запросами
- тянуть все комментарии для всего run в один ответ таблицы

### Что нужно делать

Нужно:
- строить chats/messages bulk-образом во время worker-run
- хранить результат нормализованно
- пересчитывать только затронутые `result_id`
- загружать комментарии отдельным endpoint по конкретной строке результата

## Границы первой версии

Входят:
- новый сигнал `chatActivityMatch`
- таблицы `dl_match_result_chat` и `dl_match_result_message`
- исключение `peer_id` с быстрым пересчётом
- показ всех сообщений пользователя по найденным чатам
- обновлённый экспорт без исключённых чатов

Не входят:
- повторный полный пересчёт DL-run
- история наборов исключений
- массовое исключение нескольких чатов одним действием
- полнотекстовый поиск по комментариям

## Обработка ошибок

- если загрузка сообщений для батча упала, весь run должен стать `FAILED`, потому что `chatActivityMatch` будет неполным
- если исключение `peer_id` не нашло затронутых строк, это штатный `200/204`, а не ошибка
- если пересчёт после исключения упал, backend не должен оставлять частично обновлённое состояние; нужен транзакционный пересчёт по затронутым `result_id`

## Тестирование

### Backend

- unit: `chatActivityMatch` выставляется при наличии активных чатов
- unit: исключение `peer_id` выключает `chatActivityMatch`, если активных чатов больше не осталось
- unit: результат исчезает из выдачи, если после пересчёта не осталось активных сигналов
- unit: `GET messages` возвращает все сообщения по чатам результата
- exporter test: исключённые чаты и их сообщения не попадают в `xlsx`

### Frontend

- render test: таблица показывает badge `Chat activity match`
- interaction test: кнопка `Исключить` вызывает API и удаляет строки, державшиеся только на `chatActivity`
- render test: detail-view показывает комментарии по чатам
- render test: после исключения чат и его сообщения исчезают из активного представления

## Риски

- хранение всех сообщений может заметно увеличить размер таблицы `dl_match_result_message`
- для run с большим числом активных пользователей экспорт может стать тяжёлым, поэтому надо следить за батчами чтения и стримингом workbook
- без отдельного lazy-load endpoint таблица результатов станет слишком тяжёлой, поэтому сообщения нужно грузить по строке, а не вместе со всем `GET /results`

# Дизайн: универсальный резолв идентификаторов Telegram для sync

## Контекст

Текущий сценарий `POST /telegram/sync` вызывает `client.getEntity(identifier)` напрямую из `TelegramService`.
Это работает для части публичных идентификаторов, но ломается на numeric/channel id вида `-1001157519810`.

По логу от `2026-03-13` ошибка выглядит так:

- `RPCError: 400: CHANNEL_INVALID (caused by channels.GetChannels)`
- `Could not find the input entity for {"channelId":"1157519810","className":"PeerChannel"}`

Причина: для каналов и супергрупп одного `telegramId` недостаточно. GramJS/MTProto часто требует `accessHash` или уже известный клиенту `InputEntity`.

## Цель

Сделать `syncChat` устойчивым к основным форматам входа и убрать хрупкую зависимость от прямого `getEntity()` для всех случаев.

После изменения пользователь может передать:

- `@username`
- `username`
- `t.me/name`
- `https://t.me/name`
- `https://t.me/+invite`
- `https://t.me/joinchat/...`
- `123456789`
- `-1001157519810`

Система сама определяет формат и выбирает корректный сценарий резолва.

## Не цели

- Не меняем текущую бизнес-логику сохранения участников.
- Не добавляем массовый авто-join в произвольные чаты без явной политики.
- Не переписываем весь Telegram-модуль.

## Требования

### Функциональные

- Поддержать публичные username и публичные ссылки.
- Поддержать invite-ссылки.
- Поддержать numeric id и channel/supergroup id вида `-100...`.
- Для numeric id давать контролируемую ошибку, если недостаточно данных для построения `InputEntity`.
- Сохранить совместимость текущего API `/telegram/sync`.

### Нефункциональные

- Ошибки должны быть диагностируемыми и понятными пользователю.
- Логика распознавания форматов не должна жить в `TelegramService`.
- Поведение должно покрываться unit-тестами.

## Подходы

### Вариант 1. Отдельный resolver-сервис

Ввести `TelegramIdentifierResolverService`, который:

- нормализует ввод;
- определяет тип идентификатора;
- выбирает сценарий резолва;
- возвращает готовый `entity` и метаданные резолва.

Плюсы:

- чёткое разделение ответственности;
- проще тестировать;
- проще расширять новыми форматами.

Минусы:

- появляется новый сервис и слой ошибок.

### Вариант 2. Локальные `if` в `TelegramService`

Добавить ветвления прямо в `syncChat`.

Плюсы:

- быстро внедрить.

Минусы:

- логика расползается;
- сложнее тестировать и поддерживать;
- повышается риск регрессий.

### Вариант 3. Опора на принудительное наполнение entity cache

Пытаться “познакомить” клиента с чатом через дополнительные API-вызовы и затем снова использовать `getEntity`.

Плюсы:

- может сработать без дополнительного хранения метаданных.

Минусы:

- хрупко;
- возможны побочные эффекты;
- плохо контролируется и объясняется.

### Решение

Выбрать вариант 1: отдельный `TelegramIdentifierResolverService`.

## Архитектура

### Текущий поток

`TelegramService.syncChat()`:

1. валидирует `identifier`;
2. получает Telegram client;
3. делает `client.getEntity(identifier)`;
4. передаёт результат в `TelegramChatMapper`;
5. собирает участников и сохраняет чат.

### Новый поток

`TelegramService.syncChat()`:

1. валидирует `identifier`;
2. получает Telegram client;
3. вызывает `TelegramIdentifierResolverService.resolve(client, identifier)`;
4. получает `entity`, нормализованное значение и тип резолва;
5. передаёт `entity` в `TelegramChatMapper`;
6. собирает участников и сохраняет чат вместе с метаданными резолва.

`TelegramService` остаётся orchestration-слоем и больше не знает детали форматов идентификаторов.

## Компоненты

### `TelegramIdentifierResolverService`

Ответственность:

- классификация входа;
- нормализация;
- резолв через правильный Telegram API сценарий;
- перевод инфраструктурных ошибок в доменные ошибки.

Предполагаемые методы:

- `normalize(raw: string): NormalizedTelegramIdentifier`
- `resolve(client, raw): Promise<ResolvedTelegramEntity>`

### `normalizeTelegramIdentifier` util

Выделенная утилита для синтаксического разбора входа.

Определяет один из типов:

- `username`
- `publicLink`
- `inviteLink`
- `numericId`
- `channelNumericId`
- `invalid`

### `TelegramChatRepository`

Нужно расширить репозиторий методами чтения chat metadata, достаточной для повторного построения `InputChannel`/`InputPeer` по `telegramId`.

### `TelegramChatSyncService`

Нужно сохранять metadata, достаточную для повторного резолва numeric id:

- `accessHash` для channel/supergroup;
- при необходимости `rawIdentifier` или `lastResolvedFrom`;
- уже существующие `telegramId`, `type`, `username`, `title`.

## Правила резолва

### Username и публичные ссылки

Поддержка:

- `@name`
- `name`
- `t.me/name`
- `https://t.me/name`

Поведение:

- нормализуем до username-представления;
- резолвим через `client.getEntity(normalized)`;
- успешный результат сохраняем как обычный чат.

### Invite-ссылки

Поддержка:

- `https://t.me/+hash`
- `https://t.me/joinchat/hash`

Поведение:

- извлекаем invite hash;
- сначала делаем проверочный сценарий без немого изменения состояния;
- если Telegram возвращает уже доступный чат, используем его;
- если нужен join, поведение должно быть явно определено в коде.

Рекомендуемая политика первой версии:

- не выполнять auto-join молча;
- возвращать понятную ошибку, если чат доступен только после вступления.

Это даёт безопасное и предсказуемое поведение.

### Numeric id

Поддержка:

- `123456789`
- `-1001157519810`

Поведение:

- распознаём numeric id;
- определяем, это общий numeric id или channel/supergroup формат `-100...`;
- сначала ищем чат в локальной БД по `telegramId`;
- если в БД есть нужная metadata, строим `InputChannel`/`InputPeer`;
- если metadata нет, не делаем слепой `getEntity(id)`.

Причина: именно такой слепой вызов и вызывает `CHANNEL_INVALID`.

### Ошибка недостаточных данных

Если пользователь впервые вводит только numeric/channel id, а у нас нет сохранённого `accessHash` или другого способа достроить input entity, API должен вернуть контролируемую ошибку уровня `400`.

Сообщение должно объяснять, что для первого обращения к чату нужны:

- `@username`;
- публичная ссылка;
- invite-ссылка;
- или ранее сохранённый чат.

## Модель данных

### Проблема текущей схемы

В `api/prisma/schema.prisma`:

- `TelegramUser.accessHash` уже есть;
- у `TelegramChat` поля `accessHash` сейчас нет.

Этого недостаточно для надёжного повторного резолва channel/supergroup по `telegramId`.

### Изменение схемы

Добавить в `TelegramChat` минимум:

- `accessHash String?`

Опционально, если понадобится для диагностики:

- `lastResolvedFrom String?`

Для первой версии достаточно `accessHash`.

## Ошибки и API-контракты

Нужно различать 3 класса ошибок:

1. Невалидный формат.
2. Валидный формат, но чат недоступен или не найден.
3. Валидный numeric id, но недостаточно metadata для резолва.

Пользовательское сообщение должно быть конкретным. Вместо общего:

- `Unable to resolve Telegram chat by provided identifier`

ожидаются сообщения уровня:

- `Неподдерживаемый формат идентификатора Telegram`
- `Чат Telegram недоступен по указанному идентификатору`
- `Нельзя впервые открыть чат только по numeric ID; используйте username, публичную или invite-ссылку`

Логи при этом могут сохранять исходную техническую ошибку Telegram API.

## Тестирование

### Unit

- классификация форматов идентификаторов;
- успех резолва по username;
- успех резолва по публичной ссылке;
- контролируемое поведение для invite-ссылки;
- успех резолва по numeric id из БД;
- ошибка по numeric id без `accessHash`;
- обновлённые тесты `TelegramService`.

### Интеграция

- `POST /telegram/sync` с `@username`;
- `POST /telegram/sync` с `-100...`, когда чат уже известен;
- `POST /telegram/sync` с `-100...`, когда metadata отсутствует.

## Риски

- Invite-ссылки могут требовать более аккуратной политики доступа, чем username.
- Для части старых записей `TelegramChat` metadata будет отсутствовать до первого успешного sync по публичному идентификатору.
- Миграция Prisma должна быть согласована с текущим deployment-процессом.

## Критерии успеха

- Пользователь может синхронизировать чат по `@username`, публичной ссылке и уже известному `-100...`.
- Ошибка `CHANNEL_INVALID` больше не всплывает наружу как общий fail сценария.
- Для неподдерживаемых или недостаточных данных пользователь получает точную ошибку и понятное следующее действие.

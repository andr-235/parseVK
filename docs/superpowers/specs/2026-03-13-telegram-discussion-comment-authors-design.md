# Дизайн: сбор авторов комментариев Telegram-обсуждений

## Контекст

Текущий Telegram-модуль умеет синхронизировать участников чата:

- резолвит чат по `@username`, публичной ссылке или known internal identifier;
- получает участников через `channels.GetParticipants` или `messages.GetFullChat`;
- сохраняет результат как sync участников.

Пользователю нужен другой сценарий: не все участники чата, а только люди, которые реально писали комментарии в обсуждении Telegram.

Это отдельная задача, потому что источник данных другой:

- сейчас система читает `participants`;
- для новой задачи нужно читать `messages/comments` и извлекать авторов.

## Цель

Добавить возможность собирать уникальных авторов комментариев Telegram-обсуждений в двух режимах:

- `thread`: авторы комментариев под одним конкретным постом/тредом;
- `chatRange`: авторы комментариев по discussion-чату за период или лимит сообщений.

## Не цели

- Не заменять существующий `sync участников`.
- Не добавлять массовый scraper "всей аудитории Telegram".
- Не реализовывать auto-join по invite-ссылкам.
- Не обходить ограничения Telegram.

## Требования

### Функциональные

- Добавить новый backend flow для `discussion comment authors`.
- Поддержать выбор режима:
  - `thread`
  - `chatRange`
- Поддержать входы:
  - `@username`
  - `t.me/name`
  - `t.me/c/<chatId>/<messageId>`
  - `-100...` для уже известных чатов
- Для `thread` уметь брать `messageId` из ссылки или отдельного поля.
- Для `chatRange` уметь ограничивать выборку:
  - `dateFrom/dateTo`
  - или `messageLimit`
  - опционально `authorLimit`
- Возвращать только уникальных авторов-комментаторов.

### Нефункциональные

- Не ломать текущий flow sync участников.
- Сохранять единый resolver/identifier подход там, где это возможно.
- Ошибки и helper text должны быть на русском языке.
- Логика комментариев должна быть отдельной, а не спрятанной внутри `syncChat`.

## Подходы

### Вариант 1. Отдельный discussion flow с двумя режимами

Добавляется новый endpoint и новый набор сервисов:

- `thread`
- `chatRange`

Плюсы:

- чистая архитектура;
- не ломает `sync участников`;
- обе задачи можно развивать независимо.

Минусы:

- требуется больше новых файлов и контрактов.

### Вариант 2. Расширить `syncChat` режимом `commentAuthors`

Один endpoint выполняет и sync участников, и сбор авторов комментариев.

Плюсы:

- меньше новых API route.

Минусы:

- смешиваются две разные по смыслу операции;
- усложняется `TelegramService`.

### Вариант 3. Сделать только режим одного треда

Реализовать только `thread`, а `chatRange` отложить.

Плюсы:

- проще первая версия.

Минусы:

- сразу не покрывает реальный запрос пользователя "нужна возможность выбора".

### Решение

Выбрать вариант 1: отдельный discussion flow с двумя режимами.

## Архитектура

### Новый API-сценарий

Добавляется отдельный endpoint, например:

- `POST /api/telegram/discussion-authors/sync`

Он не заменяет `POST /api/telegram/sync`, а живёт рядом.

### Входной контракт

Тело запроса:

- `identifier`
- `mode`: `thread | chatRange`
- `messageId` optional
- `dateFrom` optional
- `dateTo` optional
- `messageLimit` optional
- `authorLimit` optional

### Режим `thread`

Поведение:

- если вход `t.me/c/<chatId>/<messageId>`, `messageId` извлекается автоматически;
- если вход публичная ссылка на пост/обсуждение, `messageId` тоже извлекается;
- если приходит просто chat identifier, тогда `messageId` обязателен отдельным полем.

Результат:

- уникальные авторы комментариев для одного discussion thread.

### Режим `chatRange`

Поведение:

- `messageId` не нужен;
- история discussion-чата читается пакетно;
- сообщения фильтруются по дате или лимиту.

Результат:

- уникальные авторы комментариев из диапазона истории discussion-чата.

## Разрешение идентификаторов

Resolver для discussion flow переиспользует уже существующую логику internal/public Telegram identifiers:

- `@username`
- `t.me/name`
- `t.me/c/...`
- `-100...`

Ограничение остаётся тем же:

- unknown `-100...` или `t.me/c/...` без metadata не смогут выполнить первый bootstrap.

Для таких случаев backend возвращает понятную ошибку на русском с инструкцией:

- сначала использовать `@username`, публичную или invite-ссылку;
- после первого успешного sync можно использовать internal identifier.

## Как собирать авторов комментариев

### Ключевая разница с текущим кодом

Сейчас используется:

- `channels.GetParticipants`
- `messages.GetFullChat`

Для новой задачи нужны не `participants`, а `messages/comments`.

### Режим `thread`

Логика:

1. Резолвим канал/чат обсуждения.
2. Находим discussion thread для `messageId`.
3. Читаем сообщения комментариев в треде.
4. Из ответа строим `usersMap`.
5. Для каждого комментария извлекаем автора-пользователя.
6. Дедуплицируем по `telegramUserId`.

### Режим `chatRange`

Логика:

1. Резолвим discussion-чат.
2. Идём по истории сообщений пакетами.
3. Фильтруем по:
   - `dateFrom`
   - `dateTo`
   - `messageLimit`
4. Извлекаем только сообщения с автором-пользователем.
5. Дедуплицируем авторов.

### Что отбрасываем

- service messages;
- сообщения без автора-пользователя;
- сообщения от channel/chat sender, если целевой результат именно люди;
- deleted/empty senders.

## Модель данных и сохранение

Первая версия должна возвращать и сохранять пользователей так, чтобы не смешивать их с обычным member sync по смыслу.

Рекомендуемое поведение:

- reuse существующую таблицу Telegram users, если это уже поддерживается проектом;
- результат операции маркировать источником `discussion_comments`;
- в DTO явно показывать, что это `comment authors`, а не `chat members`.

Если полная отдельная персистентная модель окажется избыточной для первой версии, допустим минимальный вариант:

- сохранять пользователей как обычно;
- но отдельно в результате синка возвращать `mode` и `source`.

## UI / UX

### Новый режим в Telegram-интерфейсе

В Telegram-карточке или рядом с ней добавить переключение:

- `Участники`
- `Комментаторы обсуждения`

### Для `Комментаторы обсуждения`

Показывать:

- выбор подрежима:
  - `Один тред`
  - `Диапазон сообщений`
- поле `messageId`, если он не извлекается автоматически;
- поля периода или лимита сообщений для `chatRange`.

### Helper text

На русском языке:

- для `thread`: можно вставить ссылку вида `t.me/c/...`;
- для internal identifier: первый bootstrap всё ещё делается через публичный идентификатор, если metadata ещё нет.

## Ошибки

Нужны отдельные понятные ошибки:

- `Для режима одного треда требуется messageId, если его нельзя извлечь из ссылки`
- `Нельзя выполнить первый sync только по внутреннему Telegram ID...`
- `Не удалось найти discussion thread для указанного сообщения`
- `Нет доступа к истории обсуждения`
- `Неподдерживаемый формат идентификатора Telegram`

Все тексты ошибок и helper text должны быть на русском языке.

## Тестирование

### Backend

- normalizer: распознаёт `t.me/c/<chatId>/<messageId>`;
- resolver: known internal identifier проходит, unknown internal identifier возвращает bootstrap error;
- collector:
  - `thread` собирает уникальных авторов комментариев;
  - `chatRange` собирает уникальных авторов из истории;
  - service messages и non-user senders отбрасываются;
- service/API:
  - правильная маршрутизация по `mode`;
  - ошибка при отсутствии `messageId` для `thread`.

### Frontend

- отображение режима `Комментаторы обсуждения`;
- отображение полей `thread/chatRange`;
- отображение ошибок и helper text на русском языке.

### Ручная проверка

1. Вставить `t.me/c/...` на конкретный тред и получить уникальных авторов комментариев.
2. Указать discussion-чат и диапазон, получить авторов из истории.
3. Ввести unknown internal ID без metadata и получить понятную bootstrap-инструкцию.
4. Убедиться, что существующий sync участников не сломан.

## Критерий успеха

- Пользователь может собирать не всех участников чата, а именно людей, которые писали комментарии.
- Поддерживаются оба сценария:
  - один тред;
  - диапазон истории.
- Ошибки понятны и на русском языке.
- Существующий sync участников Telegram не ломается.

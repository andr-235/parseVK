# Дизайн: аудит критических ошибок проекта и план исправлений

Дата: 2026-03-16

## Контекст

Проект `parseVK` состоит из нескольких слоёв:

- `api` на NestJS с Prisma, очередями и фоновыми задачами;
- `front` на React/Vite;
- инфраструктурные и deployment-конфиги (`docker`, `deploy`, `monitoring`, CI/CD);
- операционная документация и архитектурные обзоры.

Пользователь хочет не общий code review, а подтверждаемый аудит всего проекта с фокусом только на критических ошибках и с результатом в виде готового плана исправлений.

В проекте уже есть сильные сигналы риска:

- архитектурный обзор в `docs/nestjs-architecture-review.md` с критичным замечанием про жизненный цикл Prisma;
- текущие незакоммиченные изменения в auth/session-коде фронтенда;
- наличие deployment и monitoring слоёв, которые тоже могут содержать production-critical конфигурационные ошибки.

## Цели

- Найти только подтверждённые high-priority проблемы по всему проекту.
- Подкрепить каждую проблему доказательством из кода, конфига, теста или воспроизводимой проверки.
- Отделить критические проблемы от средних и низких улучшений.
- Сформировать remediation-пакеты, из которых можно сделать отдельные безопасные этапы исправления.
- Подготовить implementation plan для следующего цикла работ.

## Не входит в текущий объём

- Исправление найденных проблем в рамках этого этапа.
- Полный список всех stylistic и medium/low приоритетных замечаний.
- Большой рефакторинг без подтверждённой критической причины.
- Переписывание архитектуры проекта “на будущее”.

## Подход

Рекомендованный вариант: риск-ориентированный аудит с доказательствами и последующим разбиением на remediation-пакеты.

Почему выбран этот вариант:

- соответствует запросу на “всё высокого приоритета сразу, но только с доказательствами”;
- снижает шум от некритичных замечаний;
- позволяет быстро перейти от анализа к исправлениям;
- даёт управляемый план внедрения вместо одного большого рискованного изменения.

## Определение критической проблемы

Проблема считается критической, если выполняется хотя бы одно условие:

- может ломать production-сценарий;
- может приводить к потере, порче или неконсистентности данных;
- создаёт реальную security или authorization уязвимость;
- может срывать деплой, откат, миграции, восстановление или фоновые задачи;
- уже подтверждённо нарушает архитектурный инвариант проекта так, что это повышает риск инцидента.

Проблемы не включаются в итоговый критический реестр, если:

- они являются вкусовыми или стилевыми замечаниями;
- не имеют подтверждаемого доказательства;
- относятся только к medium/low приоритету и не влияют на стабильность/безопасность/данные.

## Источники доказательств

Допустимые источники доказательств:

- код и конфиги репозитория;
- существующие тесты и их результаты;
- воспроизводимые локальные проверки и команды;
- уже существующие внутренние документы проекта, если они указывают на конкретные файлы, строки и сценарии отказа.

Предпочтение отдаётся прямому доказательству из runtime-кода и конфигурации. Документы без проверки кодом считаются вспомогательным сигналом, а не самостоятельным основанием.

## Области аудита

Аудит охватывает:

- `api`;
- `front`;
- инфраструктуру: `docker`, `deploy`, `monitoring`;
- CI/CD и release flow;
- конфигурацию безопасности, авторизации, работы с БД и очередями;
- операционные сценарии запуска, миграций и восстановления.

## Процесс аудита

### Этап 1. Risk sweep

Быстрый проход по репозиторию для сбора кандидатов на критические проблемы. Цель — не провести полный line-by-line review, а выделить наиболее рискованные зоны:

- auth/session lifecycle;
- Prisma и доступ к БД;
- фоновые задачи и очереди;
- deployment/rollback safety;
- миграции и startup scripts;
- внешние интеграции;
- monitoring/health checks;
- критичные frontend runtime flows.

### Этап 2. Подтверждение проблем

Для каждого кандидата есть только два исхода:

- проблема подтверждается и попадает в реестр;
- доказательств недостаточно, и проблема в итоговый список не попадает.

Подтверждение фиксируется через:

- файлы и конкретные места в коде;
- краткое описание механизма отказа;
- при необходимости команду или тест, который это подтверждает.

### Этап 3. Приоритизация

Подтверждённые проблемы сортируются по риску:

- `P0` — непосредственный риск прод-инцидента, потери данных, обхода auth или неработоспособности критичного сценария;
- `P1` — очень высокий риск, который не всегда проявляется мгновенно, но имеет серьёзный operational impact;
- всё остальное остаётся за пределами этого артефакта.

### Этап 4. Разбиение на remediation-пакеты

Проблемы группируются не по файлам, а по наборам исправлений:

- `auth/session stability`;
- `database/prisma lifecycle`;
- `deployment and rollback safety`;
- `background jobs and queue correctness`;
- `frontend runtime correctness`;
- при необходимости другие пакеты, если они действительно относятся к confirmed critical issues.

## Формат результата

Итог состоит из двух слоёв.

### Слой 1. Critical registry

Для каждой проблемы фиксируются:

- идентификатор и короткий заголовок;
- область (`api`, `front`, `infra`, `ci/cd`, `security`, `data`, `auth`);
- доказательство;
- почему это критично;
- сценарий отказа;
- минимальное рекомендуемое исправление;
- приоритет (`P0` или `P1`).

### Слой 2. Remediation plan

Для каждого remediation-пакета фиксируются:

- какие проблемы он закрывает;
- границы изменений;
- зависимые модули и файлы;
- риски внедрения;
- порядок выполнения;
- как проверять, что исправление действительно устранило проблему.

## Архитектурные ограничения для будущих исправлений

Исправления должны идти поэтапно и безопасно:

- без объединения нескольких независимых critical-зон в один большой коммит;
- с обязательной верификацией после каждого remediation-пакета;
- с сохранением совместимости production-сценариев;
- с отдельным вниманием к auth, миграциям, очередям и жизненному циклу БД-клиентов.

## Точки начального внимания

На старте аудит должен особенно проверить:

- auth refresh flow во фронтенде;
- жизненный цикл `PrismaService` и фактическое использование Prisma по модулям;
- deployment safety и поведение при частично успешном деплое;
- фоновые воркеры, очереди и обработку отказов;
- конфигурацию и startup scripts, влияющие на production boot path.

Это не ограничивает аудит только этими зонами, но задаёт первичный маршрут с наибольшей вероятностью найти `P0/P1`.

## Тестирование и подтверждение

В рамках аудита проверяются:

- существующие unit/integration/e2e тесты, если они покрывают рискованную область;
- линтеры, typecheck и build, если они помогают подтвердить проблему;
- локальные воспроизводимые команды для конфигурационных и operational сценариев.

Важно: отсутствие теста само по себе не является critical problem, если нет доказательства конкретного отказа. Но отсутствие покрытия может усиливать уже найденную критическую проблему.

## Критерии готовности

- Есть ограниченный список только подтверждённых `P0/P1` проблем.
- Для каждой проблемы приведено конкретное доказательство.
- Для каждой проблемы описан механизм риска, а не только симптом.
- Проблемы сгруппированы в remediation-пакеты.
- По remediation-пакетам подготовлен план исполнения для следующего цикла.

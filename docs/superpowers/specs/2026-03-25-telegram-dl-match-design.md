# Telegram DL Match Design

**Контекст**

В `tgmbase` уже есть импорт выгрузок DL в таблицы `DlImportBatch`, `DlImportFile` и `DlContact`. На фронте пользователь может загружать XLSX-файлы DL и просматривать импортированные контакты. Одновременно в той же базе есть таблица `user`, содержащая пользователей Telegram из `tgmbase`.

Новая потребность состоит из двух связанных сценариев:
- показать на фронте полную таблицу всех контактов DL, уже сохранённых в БД
- по кнопке запускать сопоставление этих контактов с `tgmbase.user`, сохранять результат и показывать только найденные совпадения с возможностью выгрузки в `xlsx`

**Цель**

Добавить поверх существующего DL-импорта новый модуль сопоставления, который проходит по всем `DlContact` в БД, ищет совпадения в `tgmbase.user`, сохраняет результаты отдельным набором и позволяет просматривать и экспортировать найденные совпадения без повторного пересчёта.

## Рекомендованный подход

Рекомендуется отдельный модуль сопоставления с сохранением запусков и строк результатов.

Рассматривались альтернативы:
- расчёт на лету без хранения результатов
- гибрид, где хранится только сводка запуска, а строки пересчитываются при открытии

Обе альтернативы хуже соответствуют задаче. Расчёт на лету не даёт повторного открытия и усложняет экспорт. Гибрид даёт лишнюю сложность без практической пользы. Сохранение отдельного набора результатов лучше подходит для истории запусков, экспорта и дальнейшего расширения.

## Пользовательский поток

1. Пользователь открывает страницу DL-выгрузки.
2. Страница загружает и показывает полную таблицу всех `DlContact` из БД с данными файла импорта.
3. Пользователь нажимает кнопку `Найти совпадения в tgmbase`.
4. Бэкенд создаёт новый запуск сопоставления и обрабатывает все контакты `DlContact`.
5. После завершения страница переключается в режим результатов и показывает только найденные совпадения последнего запуска.
6. Пользователь может выгрузить результат в `xlsx`.
7. Пользователь может вернуться к режиму просмотра полной таблицы DL.

## Правила сопоставления

### Обязательное правило

Основной строгий матч:
- `DlContact.telegramId` сопоставляется с `user.user_id`

Это основной и приоритетный сценарий сопоставления.

### Дополнительные сигналы

Нестрогие дополнительные матчи:
- `DlContact.username` сопоставляется с `user.username`
- `DlContact.phone` сопоставляется с `user.phone`

Эти сигналы не заменяют строгий матч по `telegramId`, а дополняют результат. Они должны отображаться отдельно и быть доступны для фильтрации в интерфейсе.

### Правила записи результата

- если найден матч по `telegramId`, результат сохраняется всегда
- если матч по `telegramId` не найден, но найден матч по `username` и/или `phone`, результат тоже сохраняется
- если совпадений нет вообще, строка результата не создаётся
- один `DlContact` может дать несколько строк результата, если найдено несколько пользователей по `username` или `phone`

## Архитектура

Фича строится как отдельный backend-модуль поверх `TgmbasePrismaService` и как расширение существующей DL-страницы на фронте.

На бэкенде модуль отвечает за:
- создание запуска сопоставления
- чтение всех `DlContact` из `tgmbase`
- поиск совпадений в `user`
- сохранение агрегатов запуска
- сохранение строк результата
- выдачу сохранённых результатов и экспорта

На фронте страница отвечает за два режима:
- режим полной таблицы DL-контактов
- режим результатов последнего запуска сопоставления

## Модель данных

Предлагается добавить две новые таблицы в `tgmbase`.

### `DlMatchRun`

Хранит запуск сопоставления:
- `id`
- `status` (`RUNNING`, `DONE`, `FAILED`)
- `contactsTotal`
- `matchesTotal`
- `strictMatchesTotal`
- `usernameMatchesTotal`
- `phoneMatchesTotal`
- `createdAt`
- `finishedAt`
- `error`

Назначение таблицы:
- хранить историю запусков
- показывать итоговую сводку
- обеспечивать повторное открытие результата
- быть источником для экспорта

### `DlMatchResult`

Хранит найденное совпадение:
- `id`
- `runId`
- `dlContactId`
- `tgmbaseUserId`
- `strictTelegramIdMatch`
- `usernameMatch`
- `phoneMatch`
- снапшот полей `DlContact`
- снапшот полей `user`
- `createdAt`

### Снапшоты данных

В `DlMatchResult` нужно хранить не только ссылки, но и снапшоты ключевых полей на момент запуска.

Из `DlContact`:
- `telegramId`
- `username`
- `phone`
- `firstName`
- `lastName`
- `fullName`
- `region`
- `originalFileName`

Из `user`:
- `user_id`
- `username`
- `phone`
- `first_name`
- `last_name`
- `premium`
- `scam`
- `bot`
- `upd_date`

Это нужно для трёх целей:
- экспорт должен работать без повторной сборки данных из нескольких таблиц
- старые результаты не должны меняться задним числом при обновлении `user`
- фронт должен получать готовую строку результата без сложного джойна на клиенте

## API

### Существующие endpoints

Существующий `GET /telegram-dl-import/contacts` нужно расширить так, чтобы он отдавал полную информацию, необходимую для таблицы всей DL-базы, а не только сокращённый набор колонок.

Существующий `GET /telegram-dl-import/files` остаётся для метаданных по импортам и возможных фильтров по имени файла.

### Новые endpoints

- `POST /telegram-dl-match/runs`
  Создаёт новый запуск сопоставления по всем `DlContact` в БД.

- `GET /telegram-dl-match/runs`
  Возвращает историю запусков с агрегатами.

- `GET /telegram-dl-match/runs/:id`
  Возвращает сводку конкретного запуска.

- `GET /telegram-dl-match/runs/:id/results`
  Возвращает только найденные совпадения по запуску.

- `GET /telegram-dl-match/runs/:id/export`
  Отдаёт `xlsx` с сохранёнными строками результата.

## Фронтенд

Фича должна жить на существующей странице DL-выгрузки, а не на отдельном экране. Это сохраняет единый workflow вокруг импорта и анализа выгрузок.

### Базовый экран

После загрузки страницы пользователь видит:
- таблицу всех `DlContact` из БД
- панель действий над таблицей

Панель действий содержит:
- `Найти совпадения в tgmbase`
- `Показать все DL`
- `Выгрузить XLSX` для активного завершённого результата

### Поведение после запуска

- на время выполнения запуска повторный старт блокируется
- интерфейс показывает состояние `RUNNING`
- после завершения таблица переключается в режим `только совпадения`
- экспорт становится доступен только для завершённого запуска

### Таблица полной DL-базы

Таблица полной базы должна показывать все `DlContact` и поддерживать хотя бы базовые фильтры по:
- `originalFileName`
- `telegramId`
- `username`
- `phone`

### Таблица результатов матчинга

Каждая строка результата показывает рядом:
- поля из DL-контакта
- поля найденного `user`
- флаги типа совпадения:
  - `ID match`
  - `Username match`
  - `Phone match`

Также для результата нужны фильтры по типу совпадения:
- только строгие `ID`
- только `username`
- только `phone`
- все

### Сводка запуска

Для завершённого запуска фронт показывает:
- сколько DL-контактов проверено
- сколько совпадений найдено
- сколько строгих матчей по `telegramId`
- сколько матчей по `username`
- сколько матчей по `phone`

## Обработка ошибок и неоднозначностей

- пустые `telegramId`, `username` и `phone` не считаются ошибкой; они просто не участвуют в соответствующем типе поиска
- если по `username` или `phone` найдено несколько пользователей, сохраняются несколько строк результата для одного `DlContact`
- если найден строгий матч по `telegramId` и одновременно дополнительные матчи по `username` или `phone`, строгий матч не теряется и должен быть явно помечен
- если запуск падает, `DlMatchRun.status` переводится в `FAILED`, ошибка сохраняется и логируется
- частично записанные результаты не должны оставаться в полуразобранном виде; для первой версии лучше считать запуск атомарным в границах логики сохранения одного run
- экспорт недоступен для `RUNNING` и `FAILED` запусков

## Производительность и режим выполнения

Первая версия может использовать обычный backend-запуск через HTTP endpoint, если текущий объём данных умеренный. Но структура должна допускать переход к background job без изменения внешнего UX:
- `POST /runs` создаёт запись запуска
- статус читается отдельно
- результат запрашивается отдельно

Это позволяет в следующей итерации при необходимости добавить прогресс, polling или websocket без переработки модели данных.

## Границы первой версии

В первую версию входят:
- показ полной таблицы `DlContact`
- запуск сопоставления по всем `DlContact` в БД
- сохранение запусков и строк результата
- просмотр только найденных совпадений
- фильтрация по типу совпадения
- экспорт результата в `xlsx`

В первую версию не входят:
- инкрементальный матчинг только новых контактов
- ручное подтверждение или отклонение совпадений
- объединение результатов нескольких запусков
- история экспортов
- live-progress по websocket

## Тестирование

### Backend unit

Нужны тесты сервиса сопоставления на сценарии:
- строгий матч по `telegramId`
- матч только по `username`
- матч только по `phone`
- несколько совпадений для одного `DlContact`
- отсутствие совпадений
- корректный подсчёт агрегатов запуска
- корректная запись `FAILED` статуса при ошибке

### Backend controller

Нужны тесты на:
- создание запуска
- получение списка запусков
- получение результатов запуска
- экспорт `xlsx`

### Frontend

Нужны тесты на:
- загрузку полной таблицы DL
- запуск матчинга по кнопке
- переключение экрана в режим результатов
- отображение флагов совпадения
- вызов экспорта завершённого запуска

## Риски

- поля `telegramId`, `username` и `phone` могут храниться в разных форматах, поэтому перед планированием реализации нужно отдельно проверить текущую нормализацию данных
- множественные совпадения по `username` и `phone` могут заметно увеличивать размер результата и требуют аккуратной структуры таблицы
- если объём `DlContact` уже большой, синхронный запуск может быть слишком тяжёлым и тогда в плане нужно сразу закладывать фоновую обработку

## Итог

Фича добавляет поверх уже существующего DL-импорта слой сохранённого сопоставления с `tgmbase.user`. Пользователь получает один рабочий экран: сначала полная таблица DL-контактов, затем по кнопке только совпадения последнего запуска и отдельный экспорт в `xlsx`. Архитектурно это остаётся компактным и расширяемым решением без лишнего перепроектирования существующего импорта.

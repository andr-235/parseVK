# Дизайн: страница "Выгрузка с ДЛ" для Telegram

Дата: 2026-03-24

## Контекст

В проекте уже есть Telegram-раздел на фронтенде, отдельная подгруппа в сайдбаре и backend-модуль для Telegram-интеграции. Также в проекте уже подключена отдельная PostgreSQL база `tgmbase` через второй Prisma schema.

Появилась новая задача: добавить в Telegram-подгруппу отдельную страницу `Выгрузка с ДЛ`, на которой оператор сможет выбрать один или несколько `xlsx`-файлов формата `groupexport_*` и загрузить их в `tgmbase`.

По согласованным требованиям:

- данные хранятся в `tgmbase`, а не в основной БД проекта;
- использовать существующие таблицы `channel`, `group`, `supergroup`, `user`, `message` не нужно;
- требуется новая предметная таблица под структуру `groupexport`;
- загрузка должна поддерживать сразу несколько файлов;
- при повторной загрузке файла новая версия должна заменять предыдущую;
- нужно сохранять имя исходного файла, чтобы было понятно, откуда пришли данные;
- данные нужно хранить не в `json`, а в явных колонках.

## Исследованный формат файлов

На основе файлов:

- `groupexport_+4AIahuktDjQxMmU6_2022-11-15.xlsx`
- `groupexport_ab3army_2024-10-15.xlsx`
- `groupexport_center_ma_2024-03-16.xlsx`

зафиксирован текущий формат:

- один лист `Sheet1`;
- первая строка содержит заголовки;
- базовые колонки присутствуют во всех примерах:
  - `Id`
  - `Username`
  - `Телефон`
  - `Имя`
  - `Фамилия`
  - `Описание`
  - `Регион`
  - `Дата`
  - `Каналы`
- в более новых файлах дополнительно встречаются:
  - `ФИО`
  - `Адрес`
  - `Вконтакте`
  - `Почта`
  - `Телеграм`
  - `Инстаграм`
  - `Viber`
  - `Одноклассники`
  - `Дата рождения`
  - второй `Username`
  - `Гео`

Это означает, что нужна нормализованная таблица с обязательными и опциональными колонками и поддержкой двух поколений формата.

## Цели

- Добавить отдельную production-страницу загрузки `xlsx` в Telegram-разделе.
- Поддержать выбор и отправку одного или нескольких файлов за один запуск.
- Сохранять имя исходного файла и статус импорта.
- Сохранять строки Excel в `tgmbase` в явных колонках.
- Поддержать замену предыдущей выгрузки новой версией файла.
- Сохранить изоляцию логики внутри отдельного backend-модуля и отдельного Prisma-клиента `tgmbase`.

## Не входит в текущий объём

- Нормализация `Каналы` в отдельную таблицу ссылок.
- Автоматический merge пользователей между разными файлами.
- Ручное редактирование импортированных строк.
- Удаление исторических выгрузок из UI.
- Импорт `.xls`, `.csv` или других форматов кроме `.xlsx`.

## Подход

Рекомендованный вариант: новая страница в Telegram-подгруппе, новый backend-модуль импорта в `tgmbase` и новый набор таблиц:

- `dl_import_batch`
- `dl_import_file`
- `dl_contact`

Почему выбран этот вариант:

- он поддерживает мультизагрузку;
- имя файла хранится на уровне файла, а не строки;
- замена предыдущей выгрузки реализуется без потери истории;
- явные колонки позволяют искать и фильтровать данные позже;
- решение не смешивает новую выгрузку с существующими таблицами `tgmbase`.

## Архитектура

### Frontend

Добавляется отдельный route, например `/telegram/dl-upload`, и новый пункт в Telegram-подгруппе сайдбара:

- `Парсинг пользователей`
- `Поиск по местным каналам`
- `Выгрузка с ДЛ`

Страница состоит из трёх зон:

1. Hero-блок с описанием формата `groupexport`.
2. Карточка выбора и загрузки нескольких `xlsx`.
3. История загруженных файлов и краткая сводка по результатам.

Для выбора файлов переиспользуется существующий компонент `FileUpload` с `multiple=true`.

### Backend

Добавляется отдельный NestJS-модуль, например `telegram-dl-import`.

Состав:

- controller для upload и чтения истории;
- service с orchestration-логикой batch/file import;
- parser service для чтения Excel и маппинга заголовков;
- repository/service layer для записи в `tgmbase`;
- DTO для upload-ответа, списка файлов и списка импортированных строк.

Логика не встраивается в существующий `telegram.service.ts`, чтобы не перегружать модуль синка Telegram API.

### Prisma и `tgmbase`

Новые таблицы добавляются в `api/prisma/tgmbase.prisma`, а работа с ними идет через существующий отдельный Prisma client для `tgmbase`.

Это позволяет:

- сохранить отдельную схему доступа к `tgmbase`;
- не смешивать модели основной БД и `tgmbase`;
- повторно использовать уже существующую инфраструктуру второго Prisma-клиента.

## Модель данных

### `dl_import_batch`

Одна запись на одно действие пользователя в UI. Если оператор выбрал пять файлов и нажал загрузить один раз, создаётся один batch.

Поля:

- `id`
- `created_at`
- `status`
- `files_total`
- `files_success`
- `files_failed`

Назначение:

- объединять несколько файлов в один пользовательский запуск;
- строить краткую сводку выполнения;
- давать базу для будущих журналов/метрик.

### `dl_import_file`

Одна запись на один `xlsx` внутри batch.

Поля:

- `id`
- `batch_id`
- `original_file_name`
- `file_hash`
- `group_slug`
- `exported_at`
- `replacement_key`
- `status`
- `rows_total`
- `rows_success`
- `rows_failed`
- `replaced_file_id`
- `created_at`
- `finished_at`
- `error`
- `is_active`

Примечания:

- `group_slug` и `exported_at` извлекаются из имени файла `groupexport_<group_slug>_<yyyy-mm-dd>.xlsx`, если шаблон соблюден;
- `replacement_key` на первом этапе фиксируется как `group_slug`;
- `replaced_file_id` указывает на предыдущую активную версию того же `replacement_key`;
- `is_active` позволяет быстро получать только актуальные версии без сложной логики на фронте.

### `dl_contact`

Одна запись на одну строку Excel.

Поля:

- `id`
- `import_file_id`
- `telegram_id`
- `username`
- `phone`
- `first_name`
- `last_name`
- `description`
- `region`
- `joined_at`
- `channels_raw`
- `full_name`
- `address`
- `vk_url`
- `email`
- `telegram_contact`
- `instagram`
- `viber`
- `odnoklassniki`
- `birth_date_text`
- `username_extra`
- `geo`
- `source_row_index`
- `created_at`

Правила:

- поля из старого формата, которых нет в новых файлах, остаются nullable;
- поля из нового формата, отсутствующие в старых файлах, также nullable;
- `channels_raw` хранится строкой без дополнительной нормализации;
- второй `Username` сохраняется отдельно как `username_extra`, чтобы не терять данные из новых файлов;
- `source_row_index` помогает разбирать ошибки импорта и сверяться с Excel.

## Правила замены

Выбранное поведение: новая загрузка заменяет предыдущую выгрузку той же сущности выгрузки.

Предлагаемая реализация:

- ключ замены фиксируется как `group_slug`, извлечённый из имени `groupexport_<group_slug>_<date>.xlsx`;
- при загрузке файла ищется текущая активная запись `dl_import_file` для этого `replacement_key`;
- создаётся новый `dl_import_file`;
- новые строки `dl_contact` пишутся только в новый `import_file_id`;
- после успешного завершения новый файл помечается `is_active = true`;
- предыдущая версия помечается `is_active = false`, а ссылка на неё сохраняется в `replaced_file_id`;
- если новый импорт падает, предыдущая активная версия остаётся активной.

Это означает, что файл `groupexport_ab3army_2024-10-15.xlsx` заменит более раннюю выгрузку `groupexport_ab3army_<другая-дата>.xlsx`, даже если полное имя файла отличается по дате.

### Инварианты целостности

На уровне БД и сервиса нужно зафиксировать:

- только одна активная запись `dl_import_file` на один `replacement_key`;
- `dl_contact` всегда принадлежит конкретному `dl_import_file`;
- заменённая запись не может снова стать активной без отдельной операции восстановления;
- batch может иметь успешный итог даже при частичных ошибках, если это отражено в счётчиках `files_success/files_failed`.

Практический вывод для миграции:

- нужен индекс по `replacement_key`;
- нужен индекс по `is_active, created_at`;
- нужен инвариант уровня БД или сервиса, который не допускает две активные записи для одного `replacement_key`.

## Пользовательский сценарий

Оператор открывает страницу `Выгрузка с ДЛ`, выбирает один или несколько файлов `groupexport_*.xlsx`, отправляет их на сервер и получает:

- статус обработки по каждому файлу;
- информацию, если файл заменил предыдущую выгрузку;
- итоговую сводку по batch;
- историю последних загруженных файлов.

Если один файл из набора невалидный, остальные продолжают импортироваться.

## UI/UX

### Верхняя часть страницы

- hero-блок с названием `Выгрузка с ДЛ`;
- короткое описание поддерживаемого формата `groupexport_*.xlsx`;
- явная подсказка, что можно выбрать несколько файлов сразу;
- пояснение, что повторная загрузка заменяет предыдущую версию файла.

### Карточка загрузки

Содержит:

- drag-and-drop или кнопку выбора файлов;
- список выбранных файлов до отправки;
- кнопку `Загрузить`;
- блок прогресса/статусов по каждому файлу.

Статусы:

- `в очереди`
- `загружается`
- `загружен`
- `заменил предыдущую выгрузку`
- `ошибка`

### История загрузок

Таблица показывает:

- имя файла;
- `group_slug`;
- дату выгрузки из имени;
- дату загрузки в систему;
- количество строк;
- статус;
- признак активной версии.

По умолчанию в истории можно показывать только последние активные файлы, а позднее при необходимости добавить фильтр истории заменённых версий.

## Контракт API

### Upload endpoint

`POST /telegram/dl-import/upload`

Формат:

- `multipart/form-data`
- поле `files[]` или `files`

Поведение:

- принимает один или несколько `xlsx`;
- создаёт один `dl_import_batch`;
- по каждому файлу создаёт `dl_import_file`;
- парсит строки и пишет `dl_contact`;
- возвращает batch summary и детальные результаты по файлам.

### История файлов

`GET /telegram/dl-import/files`

Возвращает:

- список файлов;
- статус;
- число строк;
- флаг активной версии;
- информацию о замене.

### Просмотр импортированных строк

`GET /telegram/dl-import/contacts`

Поддерживает фильтры:

- `fileId`
- `groupSlug`
- `activeOnly`
- пагинацию

Этот endpoint нужен для страницы просмотра данных и базовой верификации загрузки.

## Правила парсинга Excel

### Общие правила

- принимается только `.xlsx`;
- ожидается один лист `Sheet1`;
- первая строка трактуется как заголовок;
- пробелы по краям строк очищаются;
- пустые строки пропускаются;
- неизвестные колонки не валят импорт, а логируются как warning;
- отсутствие критически важных колонок приводит к ошибке файла.

### Маппинг заголовков

Базовый маппинг:

- `Id` -> `telegram_id`
- `Username` -> `username`
- `Телефон` -> `phone`
- `Имя` -> `first_name`
- `Фамилия` -> `last_name`
- `Описание` -> `description`
- `Регион` -> `region`
- `Дата` -> `joined_at`
- `Каналы` -> `channels_raw`
- `ФИО` -> `full_name`
- `Адрес` -> `address`
- `Вконтакте` -> `vk_url`
- `Почта` -> `email`
- `Телеграм` -> `telegram_contact`
- `Инстаграм` -> `instagram`
- `Viber` -> `viber`
- `Одноклассники` -> `odnoklassniki`
- `Дата рождения` -> `birth_date_text`
- второй `Username` -> `username_extra`
- `Гео` -> `geo`

Для дублирующегося `Username` парсер должен учитывать позицию колонки, а не только текст заголовка.

### Обязательные колонки

Минимально обязательные:

- `Id`
- `Телефон`
- `Дата`

Остальные колонки допустимо обрабатывать как опциональные.

## Ошибки и устойчивость

- ошибка одного файла не валит весь batch;
- ошибка одной строки не должна автоматически валить весь файл, если проблема локальная и строку можно пропустить;
- для каждого файла сохраняются `rows_total`, `rows_success`, `rows_failed`;
- текст ошибки сохраняется в `dl_import_file.error`;
- если импорт новой версии файла не завершился успешно, активная предыдущая версия не отключается.

### Транзакционная граница замены

Для переключения активной версии нужен атомарный сценарий:

1. создаётся новая запись `dl_import_file` в статусе `RUNNING` и `is_active = false`;
2. строки `dl_contact` пишутся в привязке к новой записи;
3. если запись строк завершилась успешно, в одной транзакции:
   - предыдущая активная версия для того же `replacement_key` переводится в `is_active = false`;
   - новая версия переводится в `is_active = true`, `status = DONE`;
   - новая версия получает `replaced_file_id` на старую;
4. если шаг 2 или 3 падает, новая версия получает `FAILED`, а старая активная версия не меняется.

Это убирает окно, в котором можно потерять активную версию при частичном падении.

## Безопасность

- endpoint доступен только авторизованным пользователям;
- бинарное содержимое файла не сохраняется в БД;
- размер файла валидируется до начала парсинга;
- в логи не пишутся чувствительные персональные данные целиком;
- строки с персональными данными не должны попадать в exception messages без необходимости.

## Производительность

- импорт выполняется пофайлово внутри одного batch;
- вставку строк лучше делать пакетно;
- UI не должен ждать детального построчного прогресса, достаточно статуса по файлу и итоговых счетчиков;
- history endpoint должен опираться на индексы по `is_active`, `created_at`, `group_slug`.

- unit-тест на чтение старого формата Excel;

### Frontend

- тест на отображение нового пункта `Выгрузка с ДЛ` в Telegram-подгруппе;

- тест на отображение ошибки по одному файлу без потери остальных результатов.

## Риски и открытые вопросы

- в формате файлов уже есть эволюция структуры, поэтому parser должен быть tolerant к отсутствию части колонок;
- поле `Каналы` иногда содержит несколько ссылок в одной ячейке с переносами строк и запятыми, поэтому на первом этапе оно хранится без разбиения;
- ключ замены зафиксирован как `group_slug`; если имя файла не соответствует шаблону `groupexport_<group_slug>_<date>.xlsx`, такой файл нужно отклонять как невалидный;
- пока предполагается, что `xlsx` всегда содержит один релевантный лист и русские заголовки в первой строке.

## Результат

После реализации оператор получит отдельную страницу в Telegram-разделе, где можно загружать несколько `groupexport`-файлов за один раз, сохранять их в `tgmbase` в явных колонках, видеть имя исходного файла и автоматически заменять предыдущую версию файла новой успешной загрузкой.
